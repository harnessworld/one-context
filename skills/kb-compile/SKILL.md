# KB Compile Skill — LLM 辅助知识编译

将外部文档（PDF/URL/本地文件/剪贴板）编译为结构化的 `knowledge/references/` 条目。
借鉴 LLM Wiki 的"知识编译"思路：先分析提取实体和关联，再生成/更新 Wiki 页面，后续可复用不再重复推理。

---

## 编译流程

```
┌─────────────────────────────────────────┐
│ Step 1: 输入解析                         │
│ URL / 本地文件 / 剪贴板内容              │
├─────────────────────────────────────────┤
│ Step 2: 增量检测                         │
│ SHA256 比对 → 已存在则跳过或标记更新     │
├─────────────────────────────────────────┤
│ Step 3: LLM 分析（两步链式）             │
│ Step 3a: 实体/关联/矛盾提取              │
│ Step 3b: 结构化 Wiki 页面生成            │
├─────────────────────────────────────────┤
│ Step 4: 与现有知识交叉检测               │
│ 检测与 standards/、playbooks/ 的关联     │
│ 检测与已有 references/ 的重复或矛盾      │
├─────────────────────────────────────────┤
│ Step 5: 生成 references/ 条目            │
│ 按模板填充，含 SHA256 标记和来源信息     │
├─────────────────────────────────────────┤
│ Step 6: 人工审核确认                     │
│ 展示编译结果，用户确认后写入             │
└─────────────────────────────────────────┘
```

---

## Step 2: 增量检测

编译前检查源文档的 SHA256 hash：

- 如果 `references/` 中已有条目的 SHA256 与本次源文档一致 → 标记 **unchanged**，跳过编译
- 如果 SHA256 不同 → 标记 **changed**，建议更新
- 如果无已有条目 → 标记 **new**，执行完整编译

SHA256 hash 存储在 references 文件的来源信息块中：

```markdown
> 来源：[Article Title](https://example.com/article)
> 作者：Author Name
> 发布日期：2026-04-15
> 收录日期：2026-04-17
> SHA256：a1b2c3d4e5f6
```

---

## Step 3: LLM 链式分析

### Step 3a: 实体提取

从源文档中提取：

| 提取项 | 说明 | 示例 |
|--------|------|------|
| 关键实体 | 核心概念、架构模式、设计决策 | PagedAttention, 连续批处理, KV Cache |
| 实体类型 | 分类标签 | 架构模式, 优化技术, 数据结构 |
| 实体关联 | 实体间关系 | PagedAttention → depends-on → KV Cache |
| 矛盾检测 | 与已有知识冲突的点 | 与 one-context 约定不一致之处 |
| 适用性评估 | 对本项目的价值判断 | 适用/部分适用/不适用 |

### Step 3b: 结构化生成

按 `templates/reference.md` 模板生成 references 条目。

---

## Step 4: 交叉检测

编译结果需与现有知识层执行交叉检测：

1. **关联检测**：提取的实体是否在 `knowledge/standards/` 或 `knowledge/playbooks/` 中有对应概念
2. **重复检测**：与 `knowledge/references/` 已有条目的主题重合度
3. **矛盾检测**：与现有约定的冲突（如 one-context 约定 vs 外部推荐做法）

输出交叉检测结果表格：

```
┌──────────────────────────────────────────────────┐
│ 交叉检测结果                                      │
├──────────────────────────────────────────────────┤
│ 关联: standards/agent-framework.md ← 实体重叠    │
│ 重复: references/openclaw-architecture.md ← 20%  │
│ 矛盾: 无                                         │
└──────────────────────────────────────────────────┘
```

---

## Step 5: 生成规范

生成的 references 条目必须遵循 `one-context-conventions.md` 的来源归属规范，并额外包含：

- **SHA256**：源文档内容的 hash，用于增量检测
- **关键实体表**：提取的核心概念
- **关联表**：实体间关系
- **交叉检测摘要**：与现有知识的关系
- **适用性评估**：对本项目的价值

具体格式见 `templates/reference.md`。

---

## Step 6: 人工审核

编译完成后**不自动写入**，而是：

1. 展示编译结果的完整 Markdown
2. 展示交叉检测结果
3. 提供操作选择：

```
📋 编译完成：llm-wiki-knowledge-compilation.md

操作选择：
  [W] 写入 knowledge/references/（推荐）
  [E] 编辑后写入
  [S] 跳过（不写入）
  [U] 更新已有条目（如已存在同主题文档）
```

只有用户明确确认后才执行写入。

---

## 使用方式

```
/kb-compile <URL或路径>     # 编译外部文档
/kb-compile --check         # 检查 references/ 中所有条目的 SHA256 是否与源一致
/kb-compile --dry-run <URL> # 预览编译结果但不写入
```

---

## 重要规则

1. **禁止自动写入**：编译结果必须经人工确认后才写入
2. **SHA256 必填**：所有编译生成的条目必须包含 SHA256 标记
3. **来源归属必填**：references/ 类型必须有完整的来源信息
4. **交叉检测必做**：编译后必须执行与现有知识的交叉检测
5. **增量优先**：已有条目 SHA256 不变则跳过，避免重复编译浪费 token
6. **工具中立**：编译内容不得绑定特定平台专有语法
7. **kebab-case 文件名**：文件名只允许 ASCII 小写字母、数字、连字符
