# KB Skill - 知识库编写智能体

知识库编写智能体，支持按规范编写知识、编辑已有知识、从外部来源导入知识。

---

## 使用方式

```
/kb                       # 显示帮助
/kb write [类型]          # 交互式编写知识（类型可选）
/kb edit <文档名>         # 编辑已有知识文档
/kb import <URL或路径>    # 从外部来源导入单篇知识
/kb check [--fix]         # 批量检查知识库文档规范性（命名、内容、格式）
```

**智能拆分**：导入时自动检测文档结构，如果发现多个独立章节，会自动询问是否拆分。

---

## 知识类型

one-context 知识库目录结构为五类，全部使用 **kebab-case 英文命名**（无前缀）：

| 类别 | 目录 | 说明 | 命名规范 |
|-----|------|------|---------|
| 规范 | knowledge/standards/ | 编码/架构/协作规范 | kebab-case 英文名，如 `agent-framework.md` |
| 操作手册 | knowledge/playbooks/ | 分步操作指南 | kebab-case 英文名，如 `add-umbrella-feature.md` |
| 提示词 | knowledge/prompts/ | 提示词模板与设计原子 | kebab-case 英文名，如 `design-atoms.md` |
| 参考资料 | knowledge/references/ | 外部文档摘要与索引 | kebab-case 英文名，如 `claudecode-architecture-patterns.md` |
| 工具文档 | knowledge/tools/ | 工具使用与配置 | kebab-case 英文名，如 `README.md` |

---

## 来源归属

所有知识文档必须在开头或正文处标注来源，格式遵循 `knowledge/standards/one-context-conventions.md`：

```markdown
> 来源：{来源名称}
> 作者：{原作者/组织}
> 发布日期：{原始发布日期}
> 收录日期：{写入 knowledge/ 的日期}
```

- **references/ 类型**：来源信息 **必填**（引用外部资料必须标明出处）
- **其他类型**：来源信息选填（原创内容可省略；从外部导入时必填）

## 核心流程

### 流程一：write - 编写新知识

```
┌─────────────────────────────────────────┐
│ Step 1: 类型选择                         │
│ (如未指定类型，交互式选择)                │
├─────────────────────────────────────────┤
│ Step 2: 交互式填空                       │
│ 逐项收集文档内容                         │
├─────────────────────────────────────────┤
│ Step 3: 去重检测                         │
│ 检查是否存在相似文档                      │
├─────────────────────────────────────────┤
│ Step 4: 生成文档                         │
│ 自动生成 kebab-case 文件名               │
├─────────────────────────────────────────┤
│ Step 5: 写入知识库                       │
│ 写入 knowledge/ 对应目录                 │
├─────────────────────────────────────────┤
│ Step 6: 询问后续操作                     │
│ 继续/提交/退出                           │
└─────────────────────────────────────────┘
```

**Step 1: 类型选择**

如果用户未指定类型，展示交互式选择菜单：

```
📝 请选择知识类型：

  [1] standards    - 编码/架构/协作规范
  [2] playbooks    - 分步操作指南
  [3] prompts      - 提示词模板与设计原子
  [4] references   - 外部文档摘要与索引
  [5] tools        - 工具使用与配置

请输入序号或类型名：
```

**Step 3: 去重检测**

扫描目标目录下的现有文档，检查是否存在标题/主题高度相似的文档。如有匹配，提示用户选择：更新现有文档 / 新建文档。

**Step 4: 自动生成文件名**

根据命名规范自动生成 kebab-case 英文文件名：
- 从用户输入的主题或标题提取关键词
- 翻译中文主题为简短英文关键词
- 使用 kebab-case 格式（小写字母、连字符分隔）
- 如有同名文件存在，追加数字后缀（如 `-2`）

**Step 5: 写入知识库**

使用 Write 工具写入文档到 `knowledge/{类型}/` 对应目录。写入前自动在文档开头或末尾补充来源归属信息（如用户未提供）。

**Step 6: 询问后续操作**

```
✅ 文档已创建：{filename}

请选择后续操作：
  [1] 继续添加其他文档
  [2] 提交当前变更
  [3] 退出（稍后手动提交）
```

### 流程三：import - 从外部来源导入（含自动拆分检测）

```
┌─────────────────────────────────────────┐
│ Step 1: 解析来源                         │
│ 支持 URL / 本地文件路径 / 剪贴板内容      │
├─────────────────────────────────────────┤
│ Step 2: 获取内容                         │
│ URL → WebFetch；本地路径 → Read；         │
│ 剪贴板 → 由用户粘贴                      │
├─────────────────────────────────────────┤
│ Step 3: 自动分析文档结构                 │
│ 检测是否包含多个独立章节                 │
├─────────────────────────────────────────┤
│ Step 4: 分支判断                         │
│ ├─ 多章节 → 询问是否拆分（Step 4a）     │
│ └─ 单主题 → 继续普通导入（Step 5）      │
├─────────────────────────────────────────┤
│ Step 4a: 拆分确认                        │
│ 展示章节列表，用户选择拆分策略           │
│ → 并行转换（主子 Agent 架构）            │
├─────────────────────────────────────────┤
│ Step 5: 类型识别 + 内容转换              │
│ 原始格式 → 知识库格式                    │
├─────────────────────────────────────────┤
│ Step 6: 去重检测与合并                   │
├─────────────────────────────────────────┤
│ Step 7: 写入知识库                       │
├─────────────────────────────────────────┤
│ Step 8: 独立质量检查 ←── 子Agent执行     │
│ 格式 | 重复 | 规范 | 解耦                │
│ （独立上下文，不共享对话历史）            │
└─────────────────────────────────────────┘
```

**Step 3-4：自动拆分检测逻辑（强制输出分析）**

> **重要**：此步骤必须**强制输出分析结果**，不得跳过。

获取文档后，**必须**执行以下分析流程：

1. **检测章节边界**（推理式检测，不限标题级别）
2. **判断独立性**（基于内容理解判断每个章节是否可以独立成文）
3. **类型预判**：对每个章节做关键词匹配，判断可能的知识类型

**强制输出分析表格**：

```
📄 文档结构分析结果

| 章节 | 是否独立 | 判断依据 | 类型推断 | 置信度 |
|-----|---------|---------|---------|--------|
| 章节A | ✅ 独立 | 完整的操作流程 | playbooks | 90% |
| 章节B | ❌ 非独立 | 辅助信息 | - | - |

结论：检测到 N 个独立章节，建议拆分导入。
```

如果检测到 **2 个以上独立章节**，继续询问用户：

```
是否拆分导入？
  [A] 全部拆分（推荐）- 并行处理，更快
  [S] 选择部分拆分
  [M] 作为单文档导入
```

### 拆分导入的并行架构

当用户选择拆分时，使用主子 Agent 并行处理：

```
┌─────────────────────────────────────────────────────┐
│ 主 Agent (Coordinator)                              │
│  • 获取文档、分析结构、用户交互                      │
│  • 汇总结果、去重检测、统一写入                      │
├─────────────────────────────────────────────────────┤
│         ↓ 派发子任务（并行执行）                     │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐   │
│  │ 子 Agent 1  │ │ 子 Agent 2  │ │ 子 Agent 3  │   │
│  │ 解析章节A   │ │ 解析章节B   │ │ 解析章节C   │   │
│  │ 类型识别    │ │ 类型识别    │ │ 类型识别    │   │
│  │ 格式转换    │ │ 格式转换    │ │ 格式转换    │   │
│  └─────────────┘ └─────────────┘ └─────────────┘   │
└─────────────────────────────────────────────────────┘
```

**边界划分**：

| 操作 | 执行者 | 原因 |
|-----|--------|------|
| 文档获取、结构分析 | 主 agent | 只需一次，需要全局视角 |
| 用户交互 | 主 agent | 避免多处询问 |
| 内容转换 | 子 agent | 可并行，加速明显 |
| 文件写入 | 主 agent | 避免竞争 |

## 子命令实现

### /kb check [--fix]

批量检查知识库文档的规范性，输出问题清单。加 `--fix` 时自动修复可修复的问题。

**检查范围**：`knowledge/` 目录下所有 `.md` 文件（排除 `README.md`）。

**检查规则**：

| # | 规则 | 严重级别 | 可自动修复 |
|---|------|---------|-----------|
| C1 | **kebab-case 文件名**：文件名必须为 kebab-case 英文格式 | ERROR | ✅ 翻译重命名 |
| C2 | **纯英文文件名**：文件名不得包含中文或其他非 ASCII 字符 | ERROR | ✅ 翻译重命名 |
| C3 | **最低内容**：正文不少于 20 行（不含来源信息和空行） | WARN | ❌ |
| C4 | **来源归属**：references/ 类型文档必须包含来源信息块 | WARN | ❌ |
| C5 | **目录归属**：文件必须位于类型对应的正确目录下 | ERROR | ✅ 移动文件 |
| C6 | **重复检测**：同目录下不得存在标题/主题高度相似的文档 | WARN | ❌ |
| C7 | **工具中立**：文档内容不得绑定特定平台或工具专有语法 | WARN | ❌ |

**输出报告格式**：

```
📋 知识库规范检查报告

扫描文件数: 42
通过: 35 | 警告: 5 | 错误: 2

━━━ ❌ ERROR (2) ━━━

  knowledge/references/推理优化.md
    [C2] 文件名包含非 ASCII 字符

  knowledge/standards/agent-framework.md
    [C5] 应位于 standards/，实际在 playbooks/

━━━ ⚠️ WARN (5) ━━━

  knowledge/references/example-ref.md
    [C4] references/ 类型缺少来源信息块

  ...
```

**--fix 自动修复**：

仅修复标记为"可自动修复"的规则：
- **C1/C2 修复**：将不规范文件名转换为 kebab-case 英文，执行 `git mv`
- **C5 修复**：将文件移动到正确目录，执行 `git mv`

修复前展示变更预览，需用户确认后执行。

## 独立质量检查

导入完成后，使用独立子 Agent 执行质量检查，不共享对话历史。

检查维度：

1. **格式检查**：kebab-case 文件名、Markdown 格式、来源归属（references/ 必填）
2. **重复检查**：同目录下是否存在同名或高度相似文档
3. **规范检查**：文件名格式、目录归属、工具中立性
4. **解耦检查**：文档可独立理解、术语有解释或链接

输出格式：
```
📄 {文件名}
质量自检：格式{✅/⚠️/❌} | 重复{✅/⚠️/❌} | 规范{✅/⚠️/❌} | 解耦{✅/⚠️/❌}
📝 详情：{各维度检查结果说明}
```

## 重要规则

1. **禁止猜测 URL**：所有外链必须由用户提供
2. **用户确认**：类型识别结果、合并策略等关键决策必须由用户确认
3. **保留历史**：编辑时保留历史记录章节
4. **批量提交**：支持批量写入后统一 commit
5. **独立质量检查**：import 完成后必须使用独立子 Agent 执行质量检查
6. **所有输出使用中文**
7. **文件名必须 kebab-case 英文**：只允许 ASCII 小写字母、数字、连字符（`-`）。中文主题必须翻译为简短英文关键词
8. **最低内容要求**：知识文档正文不得少于 20 行（不含来源信息和空行），避免空壳文档
9. **工具中立**：知识内容不得绑定特定平台或工具专有语法（如特定 SaaS 域名、专有 MCP 函数名等），保持跨平台可用
10. **来源归属**：references/ 类型必须标注来源信息；从外部导入的任何类型文档都应标注来源
