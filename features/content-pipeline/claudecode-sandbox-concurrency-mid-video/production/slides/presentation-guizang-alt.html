诚实的黑名单

默认信任，可主动收紧。

沙箱默认关闭，仅当 OMC_SECURITY=strict 时才启用。
白名单更安全但寸步难行，黑名单是在安全与可用性之间的 trade-off。

源码注释写道 "这个黑名单不是安全边界，建议配合操作系统级别的隔离。「诚实标注能力边界，比夸大宣传更能赢得工程师的信任。」"

03 / 08
File Lock

不靠 flock 的锁

为什么不用 flock flock 是"建议性锁"，恶意进程可直接无视。O_EXCL 利用文件系统存在性语义——即使对方不配合，也无法创建同名文件。
死锁自愈 锁文件内记录 PID+时间戳。超时后检查 PID 存活，进程已死则自动清理，系统自愈。

Session Lock

PID 复用防御

双重验证：不仅检查 PID 是否存活，还记录进程启动时间戳。

操作系统回收 PID 速度极快。进程 A 崩溃后 PID 被分配给全新进程 B，若只检查 PID 是否存活，会误判锁仍有效。

跨平台实现 Linux 读 /proc，macOS 用 ps，Windows 用 PowerShell —— 典型的跨平台工程实践。

乐观到悲观

"预检 → 加锁 → 重读验证"是分布式任务分配的标准三板斧。预检减少锁竞争，原子锁保证唯一性，锁内重读消除竞态条件。

07 / 08
Design Principles

四个设计原则

①
默认信任，可主动收紧
安全不应是用户体验的杀手，让用户自主选择信任等级。宽松模式可用，严格模式可选。
②
诚实声明能力边界
沙箱能防什么、不能防什么，在文档里说清楚。过度承诺比不承诺更危险。
③
不依赖合作方的诚实
用内核原子语义实现强制约束，而不是靠协议约定。O_EXCL 比 flock 更可靠。
④
分层防御
从 Python 命名空间到文件锁，从 PID 验证到 socket 检查，每层堵一个特定漏洞。
