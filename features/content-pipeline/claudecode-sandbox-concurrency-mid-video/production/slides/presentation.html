Python REPL 沙箱

黑名单机制：拦截 os、subprocess、socket 等危险模块，以及 exec、eval、open 等内置函数。

实现落点：gyoshu_bridge.py 替换 __builtins__.__import__，在导入链路上做拦截。

设计哲学（与源码自述一致）：黑名单不是安全边界；强烈建议配合操作系统级隔离。默认偏可用，用户可主动收紧策略。

Trade-off

白名单更安全，但会拖垮数据分析类工作流；黑名单是安全与可用性的折中，并需在文档里诚实写清能力边界。

第二层防御

文件存在性锁

使用 O_CREAT | O_EXCL 原子操作，而非传统 flock。内核保证只有一个进程能成功创建文件。

为什么不使用 flock?

flock 是建议性锁——其他进程可以选择不遵守。恶意或不配合的进程完全可以忽略你的锁。

核心原则

不依赖对端的自觉。利用文件系统存在性语义，即使对方不配合，也无法绕过。

第三层防御

Session 锁

PID 复用陷阱：操作系统回收 PID 的速度很快。进程A崩溃后，PID 12345 被分配给全新的进程B。

双重验证机制：不仅检查 PID 是否存活，还比对进程启动时间戳。

DOUBLE CHECK

① 读取锁文件 → 获取 PID + startTime
② 验证 PID 存活
③ 验证 startTime 匹配
④ 任一失败 → 锁已过期

总结

四个可落地的设计原则

① 默认信任，可主动收紧

不要让安全机制变成体验的负累：用可收紧的策略旋钮，把信任等级交给用户，而不是一刀切。

② 诚实声明能力边界

你的沙箱能防什么、不能防什么，在文档里说清楚。安全这件事，过度承诺比不承诺更危险。

③ 不依赖合作方的诚实

跨进程协调尽量用内核原子语义（如 O_EXCL）来实现强制约束，而不是靠协议层面的约定。

④ 分层防御

从 Python 命名空间到文件锁，从 PID 验证到 socket 类型检查，每层都在不同抽象层级堵一个特定漏洞。
