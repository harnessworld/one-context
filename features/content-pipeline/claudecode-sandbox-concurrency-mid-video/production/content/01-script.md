## 引入（约1分钟）

**小夏：** 你们公司现在是不是也在搭自己的 Agent 系统？

**智哥：** 是，基本上稍微大一点的团队都在搞这个事。

**小夏：** 那你们在做的时候有没有碰上这样的问题？让 Agent 执行代码的时候，它万一乱删乱改怎么办？

**智哥：** 这是一个。还有更具体的——多个 Agent 实例同时跑，读写同一份文件怎么避免冲突？

**小夏：** 对对对，这个很常见。还有一个，某个 worker 进程突然崩了之后，锁文件残留在那里，结果整个系统卡住。

**智哥：** 这三个问题，就是每一个做 Agent 平台的团队迟早要面对的工程难题。

**小夏：** 所以今天我们要聊的是 Claude Code —— Anthropic 自家的 AI 编程终端工具，每天都在用户电脑上执行代码、读写文件、管理并发任务。它是怎么处理这些问题的？网上几乎找不到深入分析的资料。

**智哥：** 所以我们直接读源码。今天这期，你从源码里能学到的是搭建 Agent 平台时，沙箱和存储系统该怎么设计。

## 第一层：Python REPL 沙箱——诚实的黑名单（约2分钟）

**小夏：** 好，先看第一层。Claude Code 让 AI 可以直接执行 Python 代码，做数据分析、绘图之类的，那怎么防止这段代码干坏事？

**智哥：** 答案在源码 `gyoshu_bridge.py` 里。它实现了一个黑名单式的沙箱。

**小夏：** 黑名单？不是白名单？

**智哥：** 你听我说。它拦截 `os`、`subprocess`、`socket`、`importlib` 这些危险模块，还有 `exec`、`eval`、`open` 这些内置函数。具体做法是在 Python 启动时，把 `__builtins__.__import__` 这个内置的 import 钩子替换掉——**（切换到课题片截图）** 当有代码试图 import 危险模块时，直接抛 `ImportError`。

**小夏：** 那如果遇到非危险模块呢？正常数据分析需要的包不是挺多的吗？

**智哥：** 正常通过。白名单当然更安全，但合法的数据分析场景会寸步难行。Claude Code 选了黑名单，是在安全和可用性之间做了一个 trade-off。

**小夏：** 但这个潜台词就是，它封不住的库其实还挺多的？

**智哥：** 没错。而且关键在这里——**（切换到课题片截图）** 源码里直接写了这么一句注释："这是一个黑名单，不是安全边界，建议配合操作系统级别的隔离。"

**小夏：** 哦？诚实标注自己的能力边界？

**智哥：** 对。这比夸大宣传更能赢得工程师的信任。诚实声明自己不能做到什么，本身也是一种安全设计。

**小夏：** 那回到我们自己的 Agent 平台，这块该怎么借鉴？

**智哥：** 两个启发吧。第一，沙箱不要一刀切。给用户提供「宽松/严格」两档配置，让使用你平台的人自己选择信任边界。第二，在文档里诚实声明你的沙箱能防什么、不能防什么。安全这件事，过度承诺比不承诺更危险。

## 第二层：文件锁——不靠别人配合的锁（约2.5分钟）

**小夏：** 好，接下来问题更底层了。多个进程同时读写同一份文件，怎么防冲突？

**智哥：** 这里 Claude Code 的做法很有意思——**（切换到课题片截图）** 它没有使用传统的 `flock` 文件锁，而是用了 `O_CREAT | O_EXCL`。

**小夏：** `flock` 有什么问题吗？不是 Linux 标准文件锁吗？

**智哥：** `flock` 是建议性锁。别的进程愿不愿意遵守，全凭自觉。恶意进程或者不配合的进程，完全可以无视，直接读写文件。

**小夏：** 那我的锁不就形同虚设了？

**智哥：** 对。`O_CREAT | O_EXCL` 不一样——它利用的是文件系统的存在性语义。**（切换到课题片截图）** "帮我创建这个文件，但如果文件已经存在，操作就失败。" 操作系统内核保证这个操作是原子的。两个进程同时请求，只有一个能成功创建文件。即使对方不配合、不知道你加了锁，它也创建不了同名文件。

**小夏：** 这个原理上是很漂亮，但如果拿到锁的进程自己崩了呢？锁文件不就永远留在磁盘上了？其他进程永远拿不到锁，这不就死锁了？

**智哥：** 问得好。这就是 Claude Code 的另一个设计点——**（切换到课题片截图）** 锁文件里记录了 PID 和获取锁的时间戳。超时后，系统会检查这个 PID 对应的进程是否还活着。如果进程已死，就自动删除锁文件，让系统自愈。

**小夏：** 哦，从被动等待变成主动发现。

**智哥：** 对。所以对你来说，跨进程协调优先用文件存在性做锁，而不是 `flock`。特别是在多语言混合的系统里——Python Worker、Node.js 主进程、Go 服务混在一起的时候，`flock` 在不同语言、不同操作系统上的行为经常不一致。而 `O_CREAT | O_EXCL` 是所有主流系统都支持的标准语义。

**小夏：** 而且一定要有自愈机制。

**智哥：** 对。在 Agent 平台里，子进程崩溃、OOM Kill、容器重启是常态。锁不能自动清理，一次异常就能让整个系统卡住。

## 第三层：Session 锁——PID 复用的隐藏坑（约2.5分钟）

**小夏：** 你刚才提到锁文件里记录了 PID，但这里是不是还有坑？

**智哥：** 有。PID 复用是一个隐藏得很深的坑。**（切换到课题片截图）** 操作系统回收 PID 的速度比很多人想象的要快。进程 A 持有锁之后崩溃，PID 12345 被操作系统回收，很快又被分配给了全新的进程 B。

**小夏：** 那如果系统只检查 "PID 12345 是否还活着"，岂不是误以为锁还在有效持有？

**智哥：** 完全正确。Claude Code 的解决方法是双重验证：不仅检查 PID 是否存活，还记录了进程启动时的系统时间戳。**（切换到课题片截图）** 同一个 PID 但启动时间不同，就判定为不同的进程，认为锁已经过期。

**小夏：** ——这个在不同操作系统上怎么实现？Linux 读 `/proc`，macOS 和 Windows 呢？

**智哥：** Linux 读 `/proc` 文件系统，macOS 上用 `ps` 命令，Windows 上用 PowerShell。这是典型的跨平台工程实践。

**小夏：** 还有一个细节我注意到：打开锁文件时用了 `O_NOFOLLOW` 标志。

**智哥：** 对，这个也很关键。**（切换到课题片截图）** 如果你的 Agent 平台把锁文件放在共享目录里，比如 `/tmp` 或者共享存储上，攻击者可能把锁文件路径替换成指向你系统敏感配置文件的符号链接。当锁清理逻辑试图「删除锁文件」时，实际删除的可能是你的关键文件。`O_NOFOLLOW` 就是为了堵住这个漏洞。

**小夏：** 所以，任何基于 PID 的进程存活检测都要考虑 PID 复用，这是第一个启发。

**智哥：** 对，尤其是在容器化环境或者频繁 spawn 子进程的场景下。第二个启发是，如果你的 Agent 平台涉及共享目录或者文件系统级别的操作，一定要做好符号链接攻击的防御。

## 第四层：任务级并发 Claim——从乐观到悲观（约2.5分钟）

**小夏：** 好，第四层。多个 Worker 同时竞争执行同一个任务，怎么保证一个任务只被一个 Worker 拿到？

**智哥：** 这个是分布式任务分配的精髓。Claude Code 的 Team 模式里有多个 Worker 共享一个任务列表。`findNextTask` 这个函数分三步走。**（切换到课题片截图）**

**小夏：** 是哪三步？

**智哥：** 第一步，乐观预检。不加锁，先快速看一下任务状态是不是匹配——如果状态不是 pending、owner 不是当前 worker、或者前置依赖还没完成，直接跳过。这一步目的是减少不必要的锁竞争。

**小夏：** 先筛选一遍，减少抢锁的次数？

**智哥：** 对。第二步，原子加锁。对通过预检的任务，用 `O_EXCL` 创建一个锁文件来竞争。内核保证只有一个 Worker 能成功。

**小夏：** ——那第二步拿到锁之后就安全了吗？

**智哥：** 还不行。第三步最重要：锁内重读验证。**（切换到课题片截图）** 拿到锁之后，重新读取任务状态，确认在你预检和加锁之间这段时间里，没有其他 Worker 修改了任务状态。如果状态变了，就放弃这个任务。

**小夏：** 预检和加锁之间确实有个时间窗口。

**智哥：** 对。另一个 Worker 可能在这个窗口里占领了任务。锁内重读就是在持有锁的保护下做二次确认，消除竞态条件。

**小夏：** 这三步——预检、加锁、重读——感觉可以直接复用在我们自己的多 Worker 调度系统里。

**智哥：** 这就是标准的"三板斧"：预检减少锁竞争，提升系统吞吐；原子锁保证只有一个 Worker 能拿到任务；锁内重读保证你拿到的判断不是过时的。如果你做多 Worker 的 Agent 调度系统，直接复用就行。

## 第五层：Bridge 生命周期——四层验证 + 逐级终止（约1.5分钟）

**小夏：** 最后一层，Bridge 的生命周期管理。这个 Bridge 是什么？

**智哥：** Bridge 是 Claude Code 的 Node.js 主进程和 Python 子进程之间的通信桥梁。每次连接 Bridge 的时候，Claude Code 不做「省事」的直接复用，而是做四层验证。**（切换到课题片截图）**

**小夏：** 哪四层？

**智哥：** 第一层，验证 sessionId 是否匹配——防止有人篡改了会话数据。第二层，验证 socket 路径是否是预期路径——防止路径劫持。

**小夏：** ——第三层和第四层呢？

**智哥：** 第三层，验证 PID 和进程启动时间是否一致——又是你刚才说的 PID 复用问题。第四层，验证 socket 文件类型是否正确——防止有人放一个普通文件冒充 socket。这四层验证完，才能建立通信。

**小夏：** 那如果要终止 Bridge 进程呢？

**智哥：** 信号逐级升级。**（切换到课题片截图）** 先发送 SIGINT 请求优雅退出，等 5 秒；没反应就发 SIGTERM，等 2.5 秒；还不行就用 SIGKILL 强制终止。给进程充分的清理时间，尽量减少数据损坏。

**小夏：** 还有一个设计我印象很深——Bridge 内部有一个守护线程，定期检查父进程是否还活着。

**智哥：** 对。如果父进程（Node.js）死了，Bridge 主动自杀，防止变成无人管理的孤儿进程。**（切换到课题片截图）**

## 总结（约1分钟）

**小夏：** 好，这五层都看过了。智哥能不能提炼出几个可以落地的设计原则？

**智哥：** 可以，四个原则。**（切换到课题片截图）**

第一，**默认信任，可主动收紧**。不要让安全变成用户体验的杀手，让用户自己选择信任等级。

第二，**诚实声明能力边界**。你的沙箱能防什么、不能防什么，在文档里说清楚。

第三，**不依赖合作方的诚实**。跨进程协调尽量用内核原子语义来实现强制约束，而不是靠协议层面的约定。

第四，**分层防御**。从 Python 命名空间到文件锁，从 PID 验证到 socket 类型检查，每一层都在不同的抽象层级上堵一个特定的漏洞。

**智哥：** 再见。
