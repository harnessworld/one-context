女：我听运维朋友吐槽过，他们为了跑 V4，上了八卡 A100 做张量并行，结果延迟比四卡还高，排查一整天，发现是通信开销把收益吃光了。

男：这不是个例。V4 部署资料很多，但很少有人讲清楚，哪些配置是锦上添花，哪些是火上浇油。

女：那今天要讲什么？

男：从官方 API 到生产引擎的渐进路径，四个几乎必改的推理参数，调用里三个最常见的坑。运维和开发听完都能少踩坑。

女：先说一句，为什么偏偏讲 V4？

男：跟 Kimi K2.6、GLM-5.1 比能力各有千秋，今天不比谁更聪明，只聊部署和调用角度 V4 哪里不一样。

女：你分三点说。

男：第一，MoE。V4 是混合专家，决定激活哪些专家的方式变了。同样参数规模下激活参数量更小，推理峰值显存有机会更低，但有条件，取决于 batch 和并发，后面调优再展开。

女：第二？

男：上下文更长。运维要想 max_model_len 开多大，开发要想一次塞多少 token。别误以为 128K 就是 64K 的两倍显存，真实会涨，固定开销躲不掉。PagedAttention 能缓解，但不是万能的。

女：第三？

男：量化友好。V4 在 FP8 上往往比 V3 更稳。手里是 H100、H20 这类支持 FP8 的卡，可以更敢用量化换吞吐。

女：开发听完要盯哪几个参数？

男：max_model_len、上下文怎么用、量化怎么传，调用那段都会点到。

女：进入部署，运维视角。真实产线里量化一般是叠在别的路径上的吧？

男：对。三层递进。L0，不自建，直接调官方 API，验证业务、原型、调用量还没到要上集群，就先这样。什么时候评估自建？三个信号：月费用快追上目标显卡一个月租赁成本；延迟要确定性；要锁模型版本。费用大约等于一张 A100 八十 G 月租时，通常是经济拐点，还要加上人工运维成本。

女：L1？

男：Docker 自托管。挂权重目录、暴露端口，docker run 挂上权重路径，几小时能从零跑到能响应。边界清楚：容器里若是比较裸的推理进程，没有批处理、没有请求合并，单卡 A100 八十 G 大概十到二十并发量级，看平均序列长度，几百并发不现实。

女：L1 适合什么？

男：内部工具、小团队共用，或先验证这张卡能不能把 V4 加载起来。

女：L2？

男：Docker 里跑 vLLM，不是二选一，是组合。九成九生产都这样。vLLM 强在连续批处理和 PagedAttention。TensorRT-LLM 吞吐可以更高，但构建复杂、绑 NVIDIA。建议先用 vLLM 跑通，真瓶颈再换。

女：量化什么时候叠？

男：显存装不下 FP16，或想用更少卡扛更多并发，就在 L1 或 L2 上叠。FP8、AWQ 在 HuggingFace、ModelScope 常有现成 tag，不用自己转，启动时 model 指到目录。卡支持 FP8，H100、H20 原生支持，精度损失小，优先。不支持 FP8 再看业务能不能接受四比特。摘要、分类可试 AWQ，数学、代码别硬上四比特，加卡或加大显存。

女：下面四个参数，不讲命令只讲逻辑。第一个 tensor_parallel_size 是什么？

男：每层权重矩阵按列切开，多卡各算一块再拼起来。V4 FP16 权重大概率单卡八十 G 吃不下，社区有人测大约四到八张八十 G 才稳，几乎肯定要切。

女：卡越多越好？

男：不是。两张、四张往往是甜点区，上到八张通信可能吃掉一半收益，只有极端显存不够才考虑八张。张量并行只在单节点内有效，跨节点别用这个参数，要走流水线并行。

女：pipeline_parallel_size？

男：张量并行横切，流水线竖切，不同层放不同节点，数据往下传。V4 层数多，流水线往往更明显。缺点是 bubble，前面算后面空等，可加 micro-batch 缓解但又吃显存，永远是权衡。

女：生产最常见组合？

男：单节点内横切，跨节点竖切。跨节点带宽贵，张量并行跨节点很难受。

女：KV Cache 和 PagedAttention？

男：KV 有时比权重还大。PagedAttention 把 KV 从连续大块改成离散块管理，请求结束块立刻回收。V4 上下文一长价值更大，没有它会浪费大量显存在已结束的请求上。

女：vLLM 里 max_num_seqs 怎么定？

男：太小 GPU 吃不饱，太大 KV 撑爆显存。有模型参数可粗算单请求 KV：二乘层数乘隐藏维度乘序列长度乘精度字节换算成 GB。懒得算就压测，先从十开始看 nvidia-smi，线性外推到大约八成显存，再乘零点八留余量。

女：max_model_len？

男：不是越大越好。设太大调度会过度预留，等于每个请求都按一百二十八 K 占地，实际只用四千，真实并发被挤没。先统计业务输入九十五分位再加余量，比如常态四千偶尔八千设一万二就够，别一上来写满一百二十八 K。

女：部署完要调优，总框架是什么？

男：延迟、吞吐、显存，你只能讨好其中两个，第三个一定吃亏。实时对话保延迟和显存，批任务保吞吐和显存，拿这个跟业务对齐。

女：高并发还要低延迟？

男：最难。开 continuous batching，max_num_seqs 卡在刚好不爆显存。GPU 吃没吃饱看利用率，不是光看显存，显存满了利用率六十说明 batch 还能调。

女：长上下文？

男：敌人是 KV。除 PagedAttention 可开 prefix caching，多条请求前缀相同前面 KV 不用重复算，vLLM 启动加 enable-prefix-caching。固定 system prompt、模板化多轮问答收益大。

女：卡弱或单卡？

男：量化优先，FP8 不行再上 AWQ，max_model_len 压到业务可接受最低，max_num_seqs 尽量吃满，还不行就限并发。

女：有人说 batch 越大吞吐越高？

男：误区。显存不够会 preemption，把在跑的请求换出去再换回来，成本极高吞吐反而断崖。三个信号：日志里频繁 preemption、swap blocks、recompute；利用率掉延迟猛涨；成功率还行但 P99 爆了。出现任意一条先降 max_num_seqs，别盲目扩。

女：上线第一小时自查哪三件？

男：GPU 利用率能否稳定八成以上；显存 allocated 和 reserved 差是否离谱，差太大可能是 PagedAttention 或 block 有问题；P99 延迟是否过业务 SLA。

女：开发视角，为啥运维也要听调用？

男：接口传错一个参数，前面部署全白费。

女：API 最大坑？

男：大体兼容 OpenAI 格式。第一，model 名称，官方常见 deepseek-chat、deepseek-reasoner，自托管必须跟启动里 served-model-name 一致，很多人 SDK 写 deepseek-chat 服务端不认。第二，reasoning 开关，默认常关，复杂任务要显式开，现在可能是 enable_thinking，以后改名逻辑不变，不开就是偏快偏便宜的 non-reasoning。

女：最小调用要记住什么？

男：OpenAI 客户端，base_url 指真实服务，model 对齐 served 名，要推理开 thinking，stream 解析 delta 跟 OpenAI 一样。地址对、名字对、流式别写错。

女：temperature？

男：V4 对 temperature 没以前敏感，别照搬 GPT。代码数学零点零到零点三，通用对话 RAG 零点五到零点七，创意零点八到一点二，按场景选一个档。

女：max_tokens 和 context？

男：max_tokens 管输出上限，输入上限是服务端 max_model_len，客户端改不了。长任务分段，就算支持一百二十八 K，核心信息仍建议放首尾。

女：常见 HTTP？

男：429 限流或自建队列满。503 自托管先看日志，权重路径错、GPU OOM 常见，别上来就重启。上下文超限就缩短输入或找运维评估放大 max_model_len。

女：从 V3 迁 V4 计费注意啥？

男：reasoning tokens 会在 usage 里，计入 total_tokens，不一定体现在你习惯的 completion_tokens，按 prompt 加 completion 计费可能漏。兜底用 total_tokens 计费，或把 reasoning_tokens 显式加进去。stop_reason 别写死等于 stop，reasoning 模式可能是 stop、eos、completed，用集合判断更稳。

女：给运维一句行动建议。

男：Docker 先跑通加载再上 vLLM。上线第一周：按九十五分位设 max_model_len，压测找 max_num_seqs 上限，盯 preemption。显存不够支持 FP8 就上 FP8，不支持再评估 AWQ。

女：开发？

男：跑通最小调用，核对 model、reasoning、temperature，计费用 total_tokens 别漏 reasoning。

女：正在部署的可以今晚对照三件事。

男：max_model_len 有没有按真实长度设，max_num_seqs 有没有无脑开大，GPU 利用率看起来满延迟却飙，中了任意一条今晚就能少改一轮配置。
