# 话题大纲

<!-- format: structure-v1 -->

## 整体时长

目标：~10-12 min（中视频）

## 结构流

### 00 开场（~45s）

- 钩子：DeepSeek V4 发布后，想部署的人遇到的普遍困惑——资料太多但不成体系
- 明确受众：运维工程师想知道怎么部署和调优；开发者想知道怎么调 API
- 承诺价值：看完你能画出一张自己的部署和调用决策图

### 01 认识 V4（~2min）

- V4 在同类模型中的定位（横向对比 Kimi K2.6、GLM-5.1 等）
  - 模型架构层面的关键差异（简要提及）
  - 对部署和调用产生实际影响的新特性
- 两类受众各自需要先理解的共同前置概念

### 02 部署篇 — 运维视角（~5min）

#### 02.1 部署方式概览

- 四种主要部署路径
  - 云端 API 调用（不部署，直接用）
  - Docker / 容器化部署（最轻量）
  - vLLM / TensorRT-LLM 等推理引擎部署（生产级）
  - 量化部署（FP8 / INT8 / AWQ，资源受限场景）
- 各路径适用场景速查：一张决策图

#### 02.2 核心配置项与概念解释

- 逐条讲解每个关键配置的含义：
  - `tensor_parallel_size` / `pipeline_parallel_size` — 为什么 V4 要这样切分
  - `max_num_seqs` / `max_model_len` — 与显存和吞吐的关系
  - KV Cache 管理策略 — PagedAttention 在 V4 中的表现
  - 量化路径选择 — 什么时候用 FP8，什么时候退到 INT8
- "为什么在 V4 中要这样用"：结合 V4 的架构特点解释设计意图

#### 02.3 调优思路

- 调优不是调参数，是调「延迟 × 吞吐 × 显存」三角形
- 常见调优场景：
  - 高并发低延迟
  - 长上下文（long context）
  - 资源受限（单卡 / 低端 GPU）
- 常见误区与避坑

### 03 调用篇 — 开发视角（~3min）

#### 03.1 API 调用方式

- 官方 API vs 自托管 API 的接口差异
- V4 的新参数和废弃参数

#### 03.2 传参注意事项

- `model` 字段的填写规则（v4 / v4-chat 等命名差异）
- `temperature` / `top_p` / `top_k` 在 V4 行为上的变化
- 流式输出（streaming）的启用与解析
- 长上下文传参时的 `max_tokens` 与 `context_window` 关系
- 常见 HTTP 错误码速查（429 / 503 / context length exceeded）

#### 03.3 与 V3 调用的差异

- 参数兼容性清单
- 输出格式变化（如 stop_reason、usage 字段）

### 04 总结与行动建议（~1min）

- 运维工程师：先用容器化跑通 → 再上推理引擎 → 按需加量化
- 开发者：先跑通基础 API → 关注流式与长上下文参数 → 验证输出格式变化
- 结尾 CTA：关注后续深度系列
