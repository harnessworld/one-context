开发

开发对接四大注意点

① 多轮对话结构

  • assistant reasoning_content 需显式回传
  • stream_options 合理配置，避免异常流
  • 长上下文缓存 Prefix Caching 复用 KV
reasoning 模式：默认关闭，需要时打开 enable_thinking

② 并发与资源

  • 结合机器资源与模型版本动态调整
  • 单卡 A100 80G 裸跑约 10~20 并发
  • 压测公式：(总显存−权重) / 单请求KV / 1024³ × 0.7
几百并发不可能，需要 engine 层批处理 + 多卡

③ 视觉功能（灰度）

  • 当前处于灰度阶段，关注状态更新
  • 跟进官方文档确认可用区域与版本
  • 多模态输入格式可能与纯文本不同

④ JSON Mode + 计费

  • Schema 严格匹配，Prompt 引导要到位
  • reasoning_tokens 计入 total_tokens，不计入 completion_tokens
  • 计费基数务必用 total_tokens
# 兼容 V3/V4 计费 billable = usage.total_tokens

运维

运维必查清单

① 模型文件对齐

  • 权重与 Tokenizer 配置严格一致
  • 不同框架模型 不可混用
  • vLLM / TensorRT-LLM / TGI 各用其格式
混用会导致无声错误或异常生成

② 环境参数

  • MaxModelLen 设够大，否则长上下文被 静默截断
  • max_num_seqs 压测定上限，勿无脑开
  • 按业务 95 分位长度预留余量
128K 设满 ≠ 最优，过度预留会挤占并发

③ 显存管理

  • Ingram 缓存开启
  • DevSHM 权限给够
  • 吸收注意力 + 专家并行环境变量打开
  • 防止 OOM 是第一条纪律
max_num_seqs 过大触发 preemption，吞吐反跌

④ 部署规范

  • Docker + 官方镜像 + 健康检查 + 自动重启
  • vLLM Router 做负载均衡
  • 推理强度合理配置，避免过度计算
  • 监控：GPU Util / 显存 Alloc vs Reserved / P99
