## 角色 1：视频制作达人

### 全场评级

| 段落 | 评级 | 核心判断 |
|------|------|---------|
| 00 开场 | 差 | 没有 Hook 冲击力，"真的很碎"太口语化、太软。第 10 秒观众还在判断要不要划走，你给的信息是"资料很多很碎"——这不是他们想听的第一个信息。 |
| 01 认识 V4 | 中 | 直接进入 MoE 架构，对有模型基础的人有用，但对被标题吸引来的普通运维/开发来说，开场第一段知识密度略高，需要更好的过渡句。 |
| 02.1 部署路径 | 中 | 四种路径用"并列清单"的方式呈现，长度太平均、节奏太平。每个都是"这是...适合...缺点是..."，很容易在第 3 种时疲劳。需要有一个"惊喜点"或"痛点转折"。 |
| 02.2 核心概念 | 好 | 四个概念讲解有区分度，"我不讲命令只讲逻辑"这句话很好，设定观众预期。但四个概念不宜都用同样长的篇幅，建议缩减到 3 个核心，第四个放 PPT。 |
| 02.3 调优思路 | 好 | 三角形框架非常上镜头（延迟 × 吞吐 × 显存），可以直接做一张可视化三角图。preemption 那段有"反直觉感"，适合口播。 |
| 03 调用篇 | 中 | 转场"好，运维同学的部分到这里"太平庸，需要一个更强的频道切换信号。内容本身有用但太均匀。 |
| 04 总结 | 差 | "如果你觉得有用，点个赞" 极其模板化，没有任何记忆锚点。技术类视频的结尾 CTA 最好是一个"可执行的一步行动"，而不是让用户去点赞。 |

### 3 个最严重的问题

1. **开场没有 Shocking Fact。** "资料很碎"是陈述，不是钩子。技术中视频最有效的开场通常用三种之一：① 一个反直觉的对比（"部署 V4 比部署 ChatGPT 还便宜"）；② 一个具体数字（"我跑了 40 组测试，发现 90% 的 V4 部署配置都是错的"）；③ 一个场景痛点（"花了 3 万买的显卡，结果部署 V4 只跑了 30% 的性能"）。

2. **节奏缺乏波浪感。** 02.1 四种部署路径的篇幅几乎一样（每段 4-5 行），观众在听清单时会走神。建议改为"快速带过两种 → 深入讲一种"的结构，制造节奏起伏。

3. **结尾 CTA 完全虚掉。** 最后的"点个赞"和"评论区告诉我"是 2019 年的模板，和目标受众（工程师）的心理账户不匹配。工程师的 CTA 应该是具体的 next step，比如"打开你的 vLLM 配置，对比我今天的三条检查清单"。

### 具体修改建议

- 开场第一句换成一个具体数值或场景冲击："我用 A100 单卡部署 V4，跑了 3 套配置，最快的和最慢的差了整整 8 倍。问题就出在三件事上。"
- "视频分两段" 移到认识 V4 结尾处简略带过，或者换成片头 B-roll 字幕快闪来解决。
- 02.1 缩减"官方 API"和"Docker"的篇幅到 2 句话以内，重点展开 vLLM 部署和量化路径的选择逻辑。
- 结尾去掉"点个赞"，换成「一张你今晚就能用的 3 项检查清单 + 关注」或者类似的可执行钩子。

## 角色 2：普通工程师用户（开发视角）

### 全场评级（只看 03 调用篇，兼顾 01 认识 V4）

| 段落 | 评级 | 核心判断 |
|------|------|---------|
| 01 认识 V4（扫视） | 中 | 作为开发者，我其实不太关心 MoE 架构细节。我更想知道的是：调用 V4 和调用 V3/4o/Kimi 到底哪里不一样。01 节目前置铺垫太长。 |
| 03.1 API 调用方式 | 好 | model 字段命名确实是个坑，自己部署时踩过。reasoning 开关的逻辑讲清楚了。 |
| 03.2 传参注意事项 | 中 | temperature 敏感度低的说法有意思，但"从 0.8 调到 0.6 左右"太随意了，感觉没经过非常严谨的测试。streaming SSE 偏差这个点没共鸣——但凡用 vLLM 的，SSE 不太会出问题。长上下文"信息放首尾"的技巧有收获。 |
| 03.3 与 V3 差异 | 好 | reasoning tokens 计费这个点非常实用，是确实会踩到的坑。stop_reason 那段"思考完成"和"生成完成"的区分还是有点模糊。 |

### 3 个最想听到但没讲的点

1. **SDK / 代码层面的快速示例。** 不是长篇代码，而是一个"最小可运行的调用模板"（比如 Python requests 3 行 + temperature 默认值建议）。
2. **temperature=0 是否完全 deterministic。** V4 跟 GPT 一样 temperature=0 仍可能有微小波动吗？对自动化测试场景很重要。
3. **system prompt 在 V4 上的兼容性。** V4 对 system prompt 的处理跟 V3 一样吗？有没有新增的最佳实践？

### 3 个觉得多余 / 老生常谈的段落

1. **Streaming SSE 那段（"有些引擎的流式返回格式跟标准有细微偏差"）**——在实际开发中，用 httpx / requests SSE 解析基本不会有问题，属于"说了等于没说"的警告。
2. **HTTP 429/503 那段**——对任何一个调过 API 的开发来说，429 和 503 的含义不需要解释。可以直接合并成一句话。Context length exceeded 反而更有价值，但放在错误码段落里被稀释了。
3. **"好，运维同学的部分到这里"**——这句话是给自己的转场提示，不是给观众的听觉体验。直接在视觉上切画面即可，不需要念出来。

### 具体修改建议

- reasoning / thinking 参数名不确定时，建议改成"V4 新增了推理模式开关，参数名可能随版本微调"，不要给一个似是而非的 enable_thinking 然后括号说可能不叫这个——这会让观众困惑。
- temperature 部分删掉"从 0.8 调到 0.6"这种精确数字，改成"建议比你在 GPT 上的常用值下调 0.1-0.2 测试"即可。
- HTTP 错误码部分保留 Context length exceeded，429 和 503 一句话带过。

## 角色 3：运维工程师用户

### 全场评级（只看 02 部署篇）

| 段落 | 评级 | 核心判断 |
|------|------|---------|
| 02.1 四种部署路径 | 好 | 四种路径的对比清晰，量化路径（FP8 → AWQ → 加卡）的选择逻辑是全场最有决策价值的点。但"月调用量超过一定规模"太模糊。 |
| 02.2 核心概念 | 好 | tensor_parallel_size 的"2/4张够，8张通信开销大"判断非常实用。max_model_len 不要直接设 128K 的建议直击痛点。pipeline bubble 那段对运维维护理解有要求，但讲清楚了权衡逻辑。 |
| 02.3 调优思路 | 好 | 三角形框架（延迟 × 吞吐 × 显存）我给满分，可以直接作为 PPT 核心页。preemption 机制是反常识的，有收获。 |

### 3 个最困惑或缺失的地方

1. **"月调用量超过一定规模"到底多少算一定？** 能不能给一个粗略的公式或者一个参考阈值？比如"当你的月 API 费用超过一张 A100 80G 的租赁费（约 XXX 元）时，自建开始划算"，或者给一个基于输入/输出 token 的估算 hint？
2. **没有讲部署之后的监控和告警。** 部署只是第一步，对运维来说"怎么看这个推理服务健不健康"跟"怎么部署"同等重要。至少应该提一两个关键指标：GPU 利用率 vs 显存占用（allocated vs reserved）、P99 延迟、请求队列堆积深度。
3. **Docker 部署的并发上限可以更具体。** "几十个并发可以，几百个就会崩"太粗糙。能不能给一个基于具体硬件的估算？比如"A100 80G 单卡 Docker 裸跑，实测大概能撑 10-20 个并发，取决于序列长度"。

### 3 个带走的 action item

1. 部署决策图我会截图保存，尤其是"确认 FP8 硬件支持 → 上 FP8 → 不支持则评估 4-bit 业务接受度 → 不行则加卡"这条链路。
2. max_num_seqs 的估算公式：`(总显存 - 权重) ÷ 单请求 KV Cache × 0.7`，我会回去验证这个公式在我的场景下是否成立。
3. 调优三角形——在下次团队评审时拿出来跟业务方对齐需求。实时对话保延迟，批量处理保吞吐。

### 具体修改建议

- 月调用量：给一个基于设备成本的 TTL/ROI 估算提示，比如"当你的月调用费用 ≈ 一张该型号显卡的月度租赁费时，自建是一个经济拐点"。
- 增加"部署后的第一小时检查清单"（3 条指标），作为运维的补充价值。
- Docker 并发限制给一个参考估算数字，降低模糊感。

## 追加：普通开发用户的深度补充反馈（已按反馈修改）

### 开发实操缺口

| # | 缺失内容 | 重要性 |
|---|---------|-------|
