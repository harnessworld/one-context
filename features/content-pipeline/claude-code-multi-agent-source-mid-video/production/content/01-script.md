# 工业级多 Agent 怎么做：从 Claude Code 拆隔离、消息与并行调度
（背景音乐：轻快科技感纯音乐，3秒后渐弱）

**女主持人（晓萌）**：这两年多 Agent 已经从「概念」变成了面试和落地里的硬通货——简历里敢写 Agent，面试官大概率会追问：任务怎么拆、上下文怎么隔离、并发怎么调度、消息怎么走。麻烦在于，网上教程大多是 toy demo，真正端到端的工业级样本很少，大家只能靠口口相传的经验瞎蒙。

**男主持人（阿泽）**：对，所以我们这期不收八卦、也不炒旧闻——社区里围绕 Claude Code 实现的讨论早就不新鲜了，有价值的是：**能不能把它当成一份可对照的参考实现**，把多 Agent 里最容易翻车的几块——隔离、通信、并行、成本——讲清楚。我们结合一篇写得特别细的拆解长文，把它抽象成你能拿去面试、也能搬进项目的套路。

**女主持人**：没错，今天重点是设计思想，不是逐行念代码。听完你应该能答清三件事：为什么单 Agent 不够用、CC 里常规 Subagent / Fork / Coordinator 各解决什么问题、以及五条拿来就能用的工程原则。阿泽，你先给大家说说，为什么多 Agent 突然这么火？单 Agent 不是也能干活吗？

**男主持人**：这个问题问得特别好。很多人其实没搞懂，我们为什么需要多个Agent。你想啊，经典的单Agent就是"一个LLM+一堆工具+一个循环"，你给它一个任务，它自己决定调什么工具。但一到真实场景就拉胯了。比如让它"调研React 18新特性，写个useTransition例子，再做代码评审"，单Agent会遇到三个致命问题。

**女主持人**：我猜猜，是不是上下文会爆炸？三个阶段的内容全塞进去，token根本不够用。

**男主持人**：太对了！这是第一个问题。第二个是职责混乱，它一会儿当研究员一会儿当程序员一会儿当评审员，自己都不知道自己该干嘛，经常调研到一半就去写代码，写着写着又回去查文档。第三个就是没法并发，它查文档的时候，代码就只能在那干等着。

**女主持人**：哦，我明白了。所以多Agent的思路，其实就像公司里老板带团队对吧？老板不会自己什么都干，而是把任务拆成几块，派给不同的专家。研究员负责调研，工程师负责写代码，评审员负责挑错，老板只负责看大方向、收结果。

**男主持人**：完全正确！这就是多Agent的核心思想。工业界落地的时候，一般有三种形态：父子型、平级协作型和主从型。Claude Code最牛的地方就是，它把这三种里最实用的两种都实现了，还加了一个成本优化的变种，也就是我们今天要讲的三套机制：常规Subagent、Fork Subagent和Coordinator协调者模式。

**女主持人**：听起来就很厉害。那我们先从最基础的常规Subagent开始讲吧。我看文章里说，隔离机制是多Agent设计最关键的一环，这是为什么呢？

**男主持人**：因为多Agent系统本质上就是一堆Agent共处一个进程，共享一个底层运行时。如果隔离做得不好，一个子Agent偷偷改了父Agent的状态，或者调了不该调的工具，整个系统直接就乱套了。Claude Code的隔离做得特别细，分了两个维度：工具隔离和上下文隔离。

**女主持人**：工具隔离我大概能理解，就是不能让子Agent什么工具都能用对吧？比如它不能再去派新的子Agent，不然就会无限嵌套下去，token直接花爆。

**男主持人**：对，而且Claude Code的工具隔离不是一刀切的，它做了三道权限门。第一道是所有子Agent通用的黑名单，禁止派新Agent、禁止主动问用户、禁止切换规划模式、禁止停止其他任务。第二道是自定义Agent再加一层黑名单，因为用户写的Agent没经过官方审核，得更严一点。第三道反过来，后台异步Agent走白名单，只准用读文件、搜代码这些最基础的工具。

**女主持人**：这个设计太聪明了，就像公司里不同级别的员工有不同的权限，实习生不能随便调用公司的核心系统。那上下文隔离又是什么呢？

**男主持人**：上下文隔离才是真正的精髓。很多人做隔离，要么就是完全共享父Agent的上下文，要么就是给子Agent一个全新的空上下文，这两个极端都有问题。完全共享的话，子Agent读了一个文件，父Agent就会以为自己也读过了，下次直接跳过，数据就错了。完全新建的话，用户按Ctrl+C想中止任务，子Agent根本收不到信号，变成孤儿进程一直在后台跑。

**女主持人**：那Claude Code是怎么解决这个问题的？

**男主持人**：它的思路特别巧妙：不按整体决策，而是按每个字段单独决策。比如读文件的缓存，它会克隆一份给子Agent，子怎么折腾都不影响父。写全局状态的权力直接关掉，防止两边同时改状态把界面搞花。但专门留了一个口子，允许子Agent注册后台任务，不然它起的进程就没人回收了。最后给每个子Agent一个独立ID，嵌套深度加1，超过5层就强制停止，防止失控。

**女主持人**：哇，这个细腻程度真的是工业级产品才有的。那隔离做好了之后，父子Agent之间怎么通信呢？我第一反应是直接函数调用，父Agent调个子Agent的函数，等它返回结果就行。

**男主持人**：我一开始也是这么想的，但Claude Code根本没这么做。因为函数调用是同步的，如果子Agent跑个5分钟的代码评审，父Agent就被阻塞死了，用户跟它说话都没反应。而且也没法并发，你想同时派5个子Agent调研5个模块，根本做不到。

**女主持人**：那它用的是什么方法？

**男主持人**：消息驱动。你可以把每个子Agent想象成公司里带信箱的员工。父Agent要给它布置任务，就往它信箱里扔一张字条，然后直接走人，不用等。子Agent干完活了，再把结果写成一张字条，扔回父Agent的信箱。

**女主持人**：这个比喻太形象了！那具体是怎么实现的呢？

**男主持人**：Claude Code给每个子Agent建了一份"员工档案"，里面记着它的ID、状态、信箱、结果这些信息。父Agent要发消息，就调用SendMessage工具，往目标子Agent的pendingMessages数组里追加一条消息，立刻返回。子Agent在自己的每一轮循环结束后，都会去看一眼自己的信箱，有新消息就拿出来处理。如果子Agent已经干完活停了，父Agent发消息还能自动把它唤醒，从磁盘上恢复它的对话历史继续干。

**女主持人**：那子Agent干完活怎么通知父Agent呢？

**男主持人**：这个设计更绝。它把子Agent的完成通知拼成一段XML，伪装成一条用户消息，塞给父Agent的对话历史。父Agent看到的就像用户发了一条新消息一样，直接用自己的agentic loop处理就行。为什么用XML？因为LLM对XML特别友好，不用额外教它怎么解析，而且纯文本可以直接塞进对话历史，不用搞复杂的结构化对象。

**女主持人**：太聪明了！把系统事件伪装成对话，直接复用了已有的逻辑。那如果子Agent跑的时间特别长怎么办？总不能一直占着父Agent吧？

**男主持人**：所以Claude Code做了一个自动后台化的设计。如果子Agent30秒内跑完，父Agent就在前台等，像普通工具调用一样。如果超过2分钟还没跑完，就自动转到后台，父Agent可以继续干别的。等子Agent跑完了，再通过刚才说的XML通知父Agent。

**女主持人**：这套通信机制真的太完善了，天然就支持并发。父Agent可以同时派好几个子Agent，谁先跑完谁先发通知，父Agent按顺序处理就行。

**男主持人**：没错。但这还不是最厉害的，Claude Code还有一个隐藏大招，叫Fork Subagent，专门用来解决成本问题。很多人做Agent只关心能不能跑起来，根本不关心跑起来要花多少钱，但在生产环境，成本就是能力。

**女主持人**：成本问题确实很重要。我听说Claude Code的system prompt有上万token，每次派一个新的子Agent，这些token都要重新算一遍，那成本也太高了。

**男主持人**：对，这就是多Agent最大的成本黑洞。Anthropic有个prompt缓存机制，如果API请求的前缀和之前的完全一样，这段前缀就可以走缓存，价钱只要原来的10%，延迟也大幅降低。但这个缓存的条件特别苛刻，必须是字节级完全相同，一个字、一个空格甚至一个换行不一样，都命中不了。

**女主持人**：那Fork Subagent就是为了命中这个缓存对吧？

**男主持人**：完全正确。Fork Subagent的核心思路就是，派一个和父Agent字节级完全相同的分身。它的系统prompt、用户上下文、工具池、对话历史前缀，所有会影响缓存的东西，都和父Agent一模一样。甚至它的system prompt生成函数直接返回空字符串，因为它根本不自己生成，而是直接用父Agent已经渲染好的那份字节，一个字节都不动，确保100%命中缓存。

**女主持人**：我的天，这个细节真的太工业级了，普通人根本想不到。那什么时候用Fork，什么时候用常规Subagent呢？

**男主持人**：Fork适合那种需要完整继承父Agent上下文，但又不想污染主循环的轻量任务，比如生成PR描述、做回合总结。如果是有明确专业分工的任务，比如专门搜代码、专门做规划，那还是用常规Subagent，因为它们需要定制的system prompt。而且Fork和Coordinator模式是互斥的，不能同时用。

**女主持人**：聊到Coordinator模式，这应该是Claude Code最"多Agent"的部分了吧？前面讲的都是父子结构，一次派一个子Agent，那如果有大任务需要一堆Agent同时开工怎么办？

**男主持人**：这就是Coordinator模式要解决的问题。这个模式不是默认开的，需要设置环境变量CLAUDE_CODE_COORDINATOR_MODE=1才能启用。开启之后，主Agent的角色会发生根本性变化，从什么都干的全能选手，变成一个纯协调者，只做三件事：派worker、收结果、合成答案。

**女主持人**：就像公司里的项目经理对吧？自己不写代码，只负责分配任务、整合结果。

**男主持人**：太对了。而且Coordinator模式的核心就是并行优先。它的system prompt里写了一句话，我特别喜欢："并行是你的超能力，worker全是异步的，能并行的绝不串行，多找机会一口气派一堆出去"。它可以在一次LLM回合里，同时生成多个派worker的工具调用，底层会一起并发执行。比如同时派三个worker分别调研三个模块，原来串行要10分钟的活，现在3分钟就干完了。

**女主持人**：那一个完整的任务在Coordinator模式下是怎么跑的？

**男主持人**：分成四个阶段：第一阶段是调研，多个worker并行调查代码库、理解问题；第二阶段是合成，协调者亲自读所有worker的结果，理解全局，写出具体的实现规格；第三阶段是实现，派worker按规格做具体的代码修改；第四阶段是验证，再派新的worker测试改动是否有效。

**女主持人**：为什么合成阶段必须协调者亲自做？不能让worker自己合成吗？

**男主持人**：这是一个非常重要的设计哲学：协调者必须理解，而不能转发。如果协调者只是把A worker的结果转发给B worker，那它就没有存在的价值了，worker直接跟用户对话就行了。而且协调者还要决定，遇到新任务是给老worker发消息续命，还是派个新worker。比如刚查完文件现在要改，就用老worker，因为它已经有上下文了。如果是验证这种需要新鲜眼光的工作，就必须派新worker，不能让写代码的人自己验自己。

**女主持人**：这个设计真的太符合人类团队的工作逻辑了。那Coordinator模式和常规Subagent有什么本质区别？

**男主持人**：我给你列个对比你就明白了。常规Subagent的主Agent是全能选手，子Agent默认同步执行，2分钟后才转后台，适合单个任务加临时帮手，系统形态是父子树。而Coordinator模式的主Agent是纯协调者，worker默认异步，最大化并发，适合大任务高并发拆解，系统形态是一个协调者加一堆worker的扁平结构。而且通过工具权限，worker不能再派新的worker，防止递归嵌套失控。

**女主持人**：太棒了！讲完这三套机制，我感觉对多Agent系统的理解直接上了一个台阶。那文章最后有没有总结一些可以直接复用的设计原则？

**男主持人**：有，而且每一条都能直接用到项目里，或者面试答案里。第一条，上下文隔离要按字段粒度做，不要一刀切。第二条，通信走消息，不走函数调用。第三条，工具权限要分级管控，按Agent身份配权限。第四条，缓存友好是一种架构能力，能省80-90%的成本。第五条，并行优先，协调者只合成不转发。

**女主持人**：这五条真的是字字珠玑。其实回过头看，Claude Code的多Agent系统，没有用什么特别高深的技术，都是我们平时能见到的消息队列、缓存、权限控制这些东西。但它把这些技术组合得特别好，在隔离、通信、并发、成本每一个维度都做到了极致。

**男主持人**：没错。这就是工业级产品和玩具的区别。很多人做的多Agent系统，看起来能跑，但一到生产环境就各种问题，要么状态乱了，要么并发上不去，要么成本高得离谱。这套参考实现告诉我们，好的架构不是靠炫技，而是靠对细节的打磨，对工程问题的深刻理解。

**女主持人**：说得太好了。今天我们把 Claude Code 里这套多 Agent 机制拆开了讲——隔离怎么做、消息怎么传、Fork 怎么抠缓存、Coordinator 怎么并行——外加五条可以直接复用的设计原则。希望大家听完之后，不管是面试还是做项目，都能有所收获。

**男主持人**：最后也提醒一下大家，现在AI技术发展特别快，多Agent肯定是未来的大方向。与其追各种花里胡哨的概念，不如沉下心来，好好研究一下这些工业级的实现，这才是真正能提升自己能力的东西。

**女主持人**：没错。那本期的《AI技术深一度》就到这里了，感谢大家的收听，我们下期再见！

**男主持人**：再见！
