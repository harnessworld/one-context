# 【封面】Claude Code 大型代码库最佳实践

时长：约 50s

今天我们来拆一篇 Anthropic 刚发的技术博客，标题叫 How Claude Code Works in Large Codebases。这篇文章讲的是 Claude Code 在大型代码库里到底怎么用才好使——注意，不是「能不能用」，而是「怎么配置才能用好」。因为 Anthropic 在文章里提出了一个很反直觉的核心观点：Claude Code 的能力上限，取决于你怎么配它，模型本身有多强反倒是其次的。

这个观点其实挺颠覆的。我们后面会看到，围绕这个观点，Anthropic 给出了一套完整的七层扩展架构、三个成功模式、和一套企业级三步部署方法论。内容非常扎实，我们逐层拆开来讲。

---

# 【不靠索引：Agent 式搜索】

时长：约 2min

先说第一个关键问题：Claude Code 在百万行级别的代码库里，是怎么找到自己需要的东西的？

答案出乎意料地朴素——它跟一个真人工程师拿到一个新项目时做的事情差不多：遍历文件系统、读文件、grep 搜索、追踪引用。没有什么花哨的技术，就是 Agent 式的搜索。

这跟早期 AI 编程工具走的路完全不一样。早期的方案是 RAG：先把整个代码库做 embedding，查询的时候检索相关代码块。小项目上确实没问题，但到了大型工程团队就会暴露一个致命缺陷——索引更新的速度，永远跟不上几千个工程师提交代码的速度。

想象一下：等你查的时候，拿到的可能是两周前被重命名的函数、上个 sprint 已经删掉的模块。而且更糟糕的是，没有任何提示告诉你这个结果已经过期了。你基于一个不存在的函数名去写代码，整个链路就全错了。

Claude Code 的 Agent 式搜索直接绕开了这个问题：没有 embedding 管道，没有集中式索引，每个开发者的实例直接对着最新的代码工作。你改了文件，下一秒 Claude 读到的就是最新的。

当然也有代价。因为 Agent 需要足够的起始上下文才能知道从哪里开始找。如果你让它在十亿行代码里盲搜，context window 还没开始干活就满了。所以——花时间把代码库对 Claude 配置好，会有相当可观的收益。怎么配？这就是接下来要讲的七层架构。

---

# 【七层扩展架构概览】

时长：约 1min

Anthropic 把围绕 Claude Code 构建的工具生态叫做 harness——套件。由七个扩展点组成，按构建顺序从底到顶排列：CLAUDE.md 文件、Hooks、Skills、Plugins、LSP、MCP 服务器、子 Agent。

这里有两点特别值得注意：第一，每一层都建立在前一层的基础上，顺序非常讲究，不能跳级；第二，这个架构的核心隐喻是赛马——马当然重要，但鞍具、马蹄铁、骑手的配合加在一起，对比赛结果的影响未必比马小。换句话说，模型是地板，配置才是天花板。

好，我们一层一层来看。

---

# 【第 1-2 层：CLAUDE.md + Hooks】

时长：约 2min

第一层是 CLAUDE.md 文件，也是最基础的一层。

这是 Claude 每次会话自动加载的上下文文件。用法很灵活：根目录放全局信息——我们是什么项目、用什么框架、有哪些关键约定；子目录放局部约定。Claude 在文件系统中移动的时候会自动向上遍历，逐层叠加加载。

但有一个常见的坑：塞太多东西进去反而会拖累性能。因为每个会话都会加载 CLAUDE.md，你把所有专业知识全塞进去，相当于每次对话都要先读完一整本手册，效率肯定低。那些可复用的专业知识，应该放到 Skills 里按需加载——这个后面会讲。

第二层是 Hooks。

大多数人把 Hooks 当防护栏用——「别做某事」。但 Anthropic 说，Hooks 更有价值的用法是让系统自我进化。

什么意思呢？比如一个 stop hook，在会话结束时反思发生了什么，趁 context 还热乎，提议更新 CLAUDE.md。这样 Claude 每次会话结束都能自动沉淀经验，下次就不用重复踩坑。再比如一个 start hook，根据开发者当前所在的模块，动态加载对应的团队配置。每个人进入自己负责的区域，自动获得正确的上下文。

另外，对于 lint、格式化这类确定性检查，Hooks 跑脚本比让 Claude「记住一条规则」靠谱得多。确定性的事情交给确定性工具，不确定的事情才交给 AI。

---

# 【第 3-4 层：Skills + Plugins】

时长：约 1.5min

第三层是 Skills——按需加载的专业知识包。

大型代码库有几十种不同类型的任务：安全审查、文档更新、部署配置……把所有专业知识都塞进每个会话，既不现实也不高效。Skills 的做法是渐进式披露：做安全审查时加载安全 Skill，写文档时加载文档 Skill，按需取用。

更好用的是，Skills 还可以绑定到特定目录。比如支付服务的部署 Skill 绑到支付目录下，在 monorepo 其他地方干活时就不会被自动加载——既保持专注，又不浪费 context。

第四层是 Plugins，它解决了一个老问题：好的配置容易变成部落知识。

什么意思？张三花了两周调出一套完美的 Claude 配置——哪些 Skill 好用、哪些 Hook 要配、哪些 MCP 要接。但李四入职的时候完全不知道这些。Plugin 把 Skills、Hooks、MCP 配置打成一个包，新人第一天装上就拥有和老手一样的环境。Plugin 更新也可以通过统一的 marketplace 分发到整个组织。

Anthropic 还举了一个真实的案例：一家大型零售企业造了一个 Skill，让 Claude 直接连接他们的内部数据分析平台，业务分析师不用切换工具就能拉取性能数据。推广之前，他们把这个打成了 Plugin，一键分发。

---

# 【第 5 层：LSP 精准导航】

时长：约 1min

第五层是 LSP——语言服务器协议。

你的 IDE 里大概率已经有 LSP 在跑了，提供「跳转到定义」「查找所有引用」这些功能。把这些能力暴露给 Claude，它就能按符号精确导航，区分不同语言里同名函数的差别。

没有 LSP 的话，Claude 只能靠文本匹配。在一个百万行的代码库里 grep 一个常见函数名，比如 `handle` 或者 `process`，可能返回几千个结果。Claude 要逐个打开文件去判断哪个是它要找的，context 就这么烧完了。

Anthropic 提到有一家企业软件公司，在全组织部署 Claude Code 之前，先在公司范围铺了 LSP 集成，专门为了让 C 和 C++ 的代码导航更可靠。对于多语言代码库来说，这算是回报最高的投入之一。

不过需要注意的是：LSP 不是开箱即用的。你得自己装对应语言的 code intelligence 插件和语言服务器。

---

# 【第 6-7 层：MCP 服务器 + 子 Agent】

时长：约 2min

第六层是 MCP 服务器——Claude 连接内部工具和数据源的接口。

段位最高的团队会构建 MCP 服务器，把结构化搜索暴露为 Claude 可以直接调用的工具。也有团队用 MCP 接上了内部文档系统、工单系统、或者数据分析平台。

但 Anthropic 的建议很明确：别在基础还没搭好的时候就急着上 MCP。先把 CLAUDE.md、Hooks、Skills 这些基本功做扎实再说。这就像盖楼——地基没打好就装电梯，电梯再高级楼也是歪的。

第七层是子 Agent——独立的 Claude 实例，有自己独立的 context window。

它接收一个任务，做完之后只把最终结果返回给主 Agent。有一个很实用的模式：先用一个只读的子 Agent 把某个子系统摸清楚，把发现写成文件，然后主 Agent 带着完整的上下文去做编辑。

核心价值就一句话：把探索和编辑分开做。在同一个会话里又探索又编辑，context 很容易就被撑爆了。探索阶段产生的中间信息——那些你读过但不需要保留的文件内容——通过子 Agent 天然隔离出去了。

---

# 【模式一：让代码库可读】

时长：约 1.5min

七层扩展讲完了，接下来看 Anthropic 总结的三个在成功部署中反复出现的模式。

第一个模式：让代码库对 Claude 可读。

Claude 在大型代码库里能帮多少忙，取决于它能不能找到正确的上下文。加载太多会拖累性能，加载太少就两眼一抹黑。

几个实践经验：

CLAUDE.md 文件要精简且分层。根目录只放指针和关键注意事项，细节下沉到子目录。

在子目录初始化 Claude，别从仓库根目录开始。这在 monorepo 里有点反直觉——你会担心丢掉根目录的上下文。但 Claude 会自动向上遍历加载所有 CLAUDE.md 文件，根目录的上下文不会丢。好处是工作范围被精确限定在了相关的代码区域，不会跑到十万八千里外去读不相关的代码。

测试和 lint 命令也应该按子目录配置。Claude 改了一个服务，你让它跑整个 monorepo 的测试套件？那就等着超时吧。每个子目录的 CLAUDE.md 应该写清楚该跑哪些命令。

用 .claudeignore 排除生成文件、构建产物和第三方代码。把 permissions.deny 规则提交到 .claude/settings.json，整个团队自动共享这些排除规则。

如果你的目录结构本身不够自解释，就在根目录放一个 markdown 文件，给代码库画张地图——列出每个顶层文件夹的一句话说明。

---

# 【模式二：配置要迭代】

时长：约 1min

第二个模式：随模型进化，主动维护你的配置。

这是一个很容易被忽略的点。为当前模型写的指令，在下一代模型上可能适得其反。

举个例子：一条 CLAUDE.md 规则要求 Claude 每次重构只改一个文件。在老模型上确实有效，能帮它保持专注。但新模型已经完全能做跨文件的协调编辑了，这条规则反而变成了枷锁——你花了更多钱买了更强的模型，结果被一条旧规则锁住了手脚。

同样的道理，为弥补模型弱点而写的 Skills 和 Hooks，模型一升级就可能变成多余的负担。

Anthropic 还举了另一个例子：有团队写了个 Hook，在每次文件写入时自动执行 `p4 edit`——这是 Perforce 版本控制命令。后来 Claude Code 原生支持了 Perforce 模式，这个 Hook 就完全没用了。

建议：每 3 到 6 个月做一次完整的配置审查。每次大模型发布后也值得检查一轮。如果你觉得 Claude Code 的表现到了某个瓶颈怎么也上不去——问题可能不在模型，而在你的配置没跟上。模型都已经往前跑了，你的 CLAUDE.md 可能还停在三个月前。

---

# 【模式三：得有人管】

时长：约 1min

第三个模式，也是最容易被技术团队忽略的：指定专人负责 Claude Code 的管理和推广。

技术配置做得再好，也撑不起全组织的采用。Anthropic 观察到一个共同点：推广最快的组织，在大面积开放之前，都是先有一小队人把基础设施搭好了。

有的公司是两三个工程师在 rollout 前就造好了一整套 Plugins 和 MCP。有的甚至组了专门管理 AI 编程工具的团队，一切就绪才开放访问。

效果是：开发者第一次接触 Claude Code 就能跑通。第一印象如果是「这东西不好使」，后面要翻盘就太难了。

目前承担这个职能的通常在「开发者体验」或「开发者效率」部门下。一个正在浮现的新角色叫 Agent Manager——半 PM 半工程师，专门负责管理 Claude Code 生态。规模更小的组织至少需要一个 DRI——直接责任人：一个人负责配置、权限策略、Plugin 管理、CLAUDE.md 规范，并且有拍板权。自底向上的采用能激发热情，但缺了组织层面的收敛，好用的实践就会变成部落知识，采用率到某个点就上不去了。

---

# 【治理先行 + 三步部署】

时长：约 1.5min

聊完了三个模式，再说说大组织里绕不开的问题——治理。

尤其是在受监管行业的公司里，治理问题会更早出现。谁控制哪些 Skills 和 Plugins 可用？怎么防止几千个工程师各自造轮子？AI 生成的代码怎么走和人类代码相同的审查流程？

Anthropic 的建议是从小开始：一组预定义的已批准 Skills、必须的代码审查流程、有限的初始访问权限。随着信心增长再逐步放开。

最顺畅的部署，来自那些早期就成立了跨职能工作组的组织：工程、信息安全、治理的代表坐到一起，共同定义需求，一起画路线图。不是等技术问题全解决了再谈治理，而是从一开始就并行推进。

基于这些观察，Anthropic 把企业级部署分为三个阶段：

第一阶段是基础设施阶段——小团队先搭好工具链、Plugins、MCP，把地基打好。这个阶段不要急着开放给全组织，核心是先把七层架构的前几层跑通。

第二阶段是试点阶段——有限的初始访问，配上已经定义好的审批流程。选几个有代表性的团队试用，收集反馈，调整配置。

第三阶段是规模阶段——在已经建立的治理体系和约定基础上，再大面积推广。这时候 Plugins 已经成型，MCP 服务器已经稳定，治理流程已经跑通，推广起来才不会翻车。

---

# 【收尾：适用边界】

时长：约 30s

最后说一下适用边界。

Claude Code 设计的前提是常规的软件工程环境：工程师是主要贡献者、仓库用 Git、目录结构标准。大多数大型代码库都符合这个前提。

但也有一些非常规场景需要额外配置：带大量二进制资产的游戏引擎、非 Git 版本控制的环境、非工程师往代码库里贡献内容的情况。Anthropic 说这些场景的实践指南正在路上，后续会持续更新这个系列。

总结一下今天的内容：模型能力是地板，配置质量才是天花板。七层扩展，三步部署，先搞基础再上规模，有人负责，定期翻新。说不上多新鲜的道理，但 Anthropic 是第一个把企业级部署的经验写得这么系统并分享出来的。
