## 整体定位

中视频（8-12min），深度解析 Anthropic 博客文章《How Claude Code Works in Large Codebases: Best Practices and Where to Start》。核心论点：模型能力是地板，配置质量才是天花板。

## 话题拆分

### 1. 开场：为什么大型代码库是 AI 编程工具的真正考验（~1min）

- 大型代码库的范畴：百万行 monorepo、十几年遗留系统、几十个仓库微服务
- 涵盖 C/C++/Java/PHP 等"不常联想到 AI 编程"的语言
- 引出核心观点：Claude Code 的能力上限取决于你怎么配它

### 2. 不靠索引：Agent 式搜索 vs RAG（~2min）

- Claude Code 怎么找代码：遍历文件系统、读文件、grep、追踪引用——跟真人工程师一样
- 不需要预先构建索引，不需要上传代码到服务器
- RAG 的致命缺陷：索引更新速度跟不上几千工程师提交代码的速度
- 对比图：过期索引 vs 实时搜索
- 代价：需要足够的起始上下文，否则 context window 就满了

### 3. 七层扩展架构（harness）：配置比模型重要（~4min）

- 核心比喻：赛马——马重要，但鞍具、马蹄铁、骑手配合同样重要
- 逐层讲解（从底到顶）：
  1. **CLAUDE.md**：每次会话自动加载的上下文文件，分层叠加，内容要克制
  2. **Hooks**：不只是防护栏，更让系统自我进化（stop hook 提议更新 CLAUDE.md，start hook 动态加载团队配置）
  3. **Skills**：按需加载的专业知识包，渐进式披露，可绑定到特定目录
  4. **Plugins**：打包 Skills + Hooks + MCP 配置，解决"部落知识"问题，新人第一天就有老手环境
  5. **LSP**：给 Claude 和 IDE 一样的导航能力（跳转定义、查找引用），对多语言代码库回报最高
  6. **MCP 服务器**：连接内部工具和数据源，但别在基础没搭好时急着上
  7. **子 Agent**：独立 context window，探索和编辑分开做，避免撑爆 context

### 4. 三个成功模式（~3min）

- **模式一：让代码库对 Claude 可读**
  - CLAUDE.md 精简且分层
  - 在子目录初始化 Claude（反直觉但有效）
  - 测试/lint 命令按子目录配置
  - .claudeignore 排除生成文件
  - 给代码库画张地图

- **模式二：配置要迭代，随模型进化**
  - 旧配置拖累新模型的例子（单文件重构规则 → 束缚跨文件编辑能力）
  - Perforce hook → Claude Code 原生支持后变多余
  - 建议 3-6 个月做一次配置审查

- **模式三：得有人管**
  - 推广最快的组织都是先搭好基础设施再开放
  - 第一印象决定采用率
  - Agent Manager：半 PM 半工程师的新角色
  - DRI：至少需要一个直接责任人

### 5. 治理先行与三步部署（~1.5min）

- 治理问题：谁控制 Skills/Plugins？AI 生成代码怎么走审查？
- 从小开始：预定义已批准 Skills → 代码审查流程 → 逐步放开
- 三步部署：基础设施 → 试点 → 规模化

### 6. 收尾：适用边界与下一步（~0.5min）

- 适用：常规软件工程环境（Git、标准目录结构、工程师为主要贡献者）
- 不适用：大量二进制资产、非 Git 版本控制、非工程师贡献内容
- 系列预告：超大规模文件系统、非 Git 版本控制的实践指南在路上
