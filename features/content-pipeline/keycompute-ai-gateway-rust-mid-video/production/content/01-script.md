## 开场（~1min）

<!-- PPT: 标题页 + 多 Provider 痛点图 -->

**小夏：** 公司五个业务线，有的用 OpenAI、有的用 Claude，API Key 分散在十几个仓库里，成本完全没法统计——上个月 OpenAI 账单涨了 40%，却说不出是哪个业务用的。这种问题你怎么解？

**智哥：** 网上开源网关不少，但两个硬伤——很多用 Python 或 Node.js 写的，高并发撑不住；而且几乎没有方案把个人 PC 算力也纳入调度。

**小夏：** 今天要聊的 KeyCompute 把这两个一起解决了？

**智哥：** 对。全链路 Rust，从路由到执行到计费到节点调度全部自研，而且真的能让笔记本变成算力节点。先从大图开始。

## 一、架构总览与设计哲学（~2min）

<!-- PPT: 18 crate 目录树 + 三原则卡片 -->

**智哥：** KeyCompute 把系统拆成了 18 个独立编译的模块——在 Rust 里叫 crate，类似 Java 的 module 或 Go 的 package。每个 crate 承担单一职责：llm-gateway 负责执行、keycompute-routing 负责路由、keycompute-billing 负责计费。

**小夏：** 拆 18 个好处是什么？

**智哥：** 编译隔离。改计费逻辑不用重编译路由，改前端后端一行不碰。Rust 编译慢，这种隔离是关键优化。但更重要的不是怎么拆，而是拆完之后的约束——三个核心原则。

**小夏：** 哪三个？

**智哥：** 第一，架构约束——只有 llm-gateway 能发上游请求，路由只读，计费后置，编译期就锁死了。Rust 的模块可见性机制保证其他模块想发请求都发不了。

**小夏：** 但 Gateway 挂了不是单点瓶颈吗？

**智哥：** 好问题。Gateway 只做转发和流式处理，无状态，水平扩展就行。真正的瓶颈在于后面的路由和计费，而这两个被设计成了只读和后置，反而不会卡。

**小夏：** 第二个原则？

**智哥：** 后置计费。不预扣余额，不阻塞请求，只在流结束才结算。这个后面展开。

**小夏：** 第三个？

**智哥：** 类型驱动。通过 keycompute-types 这个共享模块，统一了全系统的数据流类型——RequestContext、ExecutionPlan、StreamEvent——模块间的契约在编译期可验证。

<!-- PPT: 七阶段启动生命周期图 -->

**小夏：** 它的启动和关闭也很有意思？

**智哥：** 启动七阶段严格有序，每阶段失败立刻终止，不留半死不活的服务。值得一提的是加密初始化——API Key 用 AES 加密存储，密钥写入后不可更换，安全的代价。关闭时同时监听 SIGINT 和 SIGTERM，等正在处理的流式请求自然结束再退出——你不想因为一次部署更新让正在生成的回复断掉。

## 二、LLM Gateway：唯一执行层（~2.5min）

<!-- PPT: GatewayExecutor 架构图 + ExecutionPlan 结构体 + 流式背压示意图 -->

**小夏：** 你说 Gateway 是"唯一执行层"，这个唯一体现在哪？

**智哥：** 如果 KeyCompute 是一个操作系统，GatewayExecutor 就是它的系统调用层——所有对上游 LLM Provider 的 HTTP 请求都必须经过这里，别的模块没有这个权限。

**小夏：** 请求失败了怎么办？

**智哥：** 路由引擎会生成一个执行计划——一个 primary target 加若干 fallback target。Executor 按顺序尝试，primary 失败就试 fallback，全部失败才报错。这叫 Retry-Fallback 链。

**小夏：** 这跟一般的重试有什么区别？

**智哥：** 一般重试是同一个地址再试几次。Fallback 是换一个完全不同的 Provider 或账号。比如 primary 是 OpenAI，fallback 可能是 DeepSeek。失败不是终点，是切换通道的信号。

<!-- PPT: tokio::spawn + mpsc::channel(100) 背压代码片段 -->

**小夏：** 流式传输这块呢？

**智哥：** 一个精妙的设计——execute 方法立刻返回一个数据接收端，实际的转发放在后台异步任务里。

**小夏：** 为什么要立刻返回？

**智哥：** 避免死锁。假设缓冲区容量 100，上游高速推数据，handler 还没开始消费就填满了，发送端被挂起；但 handler 要等 execute 返回才能拿到接收端去消费——死锁。把转发放到独立任务里，让生产和消费并行运转，问题就解决了。

<!-- PPT: tiktoken o200k_base + Provider Usage 双重计量机制 -->

**小夏：** Token 计量怎么做？

**智哥：** 双重保险。流式过程中用 OpenAI 官方分词器实时估算 token 数；但流结束时，Provider 会报告精确的用量数据，系统以 Provider 报告的为准做最终修正。先估算后修正。

**小夏：** 为什么不用"字符数除以 4"？

**智哥：** 中文误差能到 50%。用官方分词器保证跟 OpenAI 计费一致，词表全局只加载一次不浪费。

**小夏：** Gateway 这块对我们做 AI 网关的启发？

**智哥：** 三个。第一，设一个唯一的请求出口，审计和监控集中管理。第二，Fallback 要跨 Provider 跨账号，不要在同一个地方反复重试。第三，Token 计量用官方分词器，不要图省事用粗估。

## 三、两层路由引擎（~2min）

<!-- PPT: 四维评分公式 + 权重常量 + 延迟分档表 + 冷却流程图 -->

**小夏：** Gateway 只负责执行，谁来决定用哪个 Provider？

**智哥：** RoutingEngine，系统的"大脑"。关键约束是——路由引擎只读无副作用，不修改任何状态，只基于快照做决策。简单可预测，不和写操作争锁。

**小夏：** 两层怎么分？

**智哥：** Layer1 对 Provider 排序，用四维加权评分：成本 30%、延迟 25%、成功率 25%、健康度 20%。所有指标统一为"越高越差"，分数越低越优先——统一方向性，避免混合正负指标的逻辑错误。

**小夏：** 延迟评分做了特殊处理？

**智哥：** 对，分档制——小于 100 毫秒 10 分，100 到 300 毫秒 30 分，300 到 1 秒 60 分，大于 1 秒 90 分。不是线性映射，更符合体感：100 毫秒和 200 毫秒差别不大，但 500 毫秒和 1.5 秒的体感差距巨大。

**小夏：** Layer2 呢？

**智哥：** 在排好序的 Provider 里选最优账号。三个因素：租户隔离——不同租户用不同账号池；模型匹配——账号必须支持请求的模型；暂时冷却——最近失败的账号暂停调度。

**小夏：** "暂时冷却"跟熔断有什么区别？

**智哥：** 不一样。熔断是完全断开、需要人工介入恢复；冷却是自动的——账号连续失败后暂时不可用，等冷却时间过了自动恢复。不是永久封禁，是暂时惩罚。

**小夏：** 路由引擎本身是只读的，健康数据哪来的？

**智哥：** 闭环。路由读快照决策，Gateway 执行后记录成功或失败，健康状态更新，影响下次路由。路由不写，但数据一直被执行层更新。Provider 连续失败时叠加 100 分惩罚，被排到队尾但不被剔除——保留自我恢复的可能性。

**小夏：** 启发？

**智哥：** 路由和执行要分离，路由只读可预测。评分指标统一方向。冷却不是永久封禁，给恢复机会。

## 四、Provider 适配器接口抽象（~1.5min）

<!-- PPT: ProviderAdapter trait 签名 + HttpTransport 依赖注入 + 6 个 Provider 列表 -->

**小夏：** 聊完执行和决策，接下来看扩展性。Provider 适配器是怎么设计的？

**智哥：** 用接口抽象——类似 Java 的 Interface 或 Go 的 interface——定义了所有 Provider 的统一契约，包括并发安全要求。

**小夏：** 加新 Provider 麻烦吗？

**智哥：** 零修改扩展。创建一个新模块，实现这个接口，在 Gateway 构建时注册就行。核心代码一行不改。

**小夏：** 测试呢？

**智哥：** 接口的第一个参数是抽象的 HTTP 传输层，不是具体客户端。测试时注入 mock，完全不依赖网络。还有一个隐藏好处——传输层之下有个 HttpProxy 层，统一管理上游连接、多代理出口、连接池复用，Provider 适配器完全不感知这些基础设施。

**小夏：** 已支持哪些 Provider？

**智哥：** 六个。OpenAI 兼容接口、Claude、Gemini、DeepSeek、Ollama 本地模型、vLLM 自托管。

## 五、Node Gateway：PC 变算力节点（~2min）

<!-- PPT: Pull-based 工作流五步图 + 容错参数表 + NodeCapabilityIndex 查询条件 -->

**小夏：** 适配器讲了云端 Provider，但 KeyCompute 还有个创新——让个人电脑变成算力节点？

**智哥：** 对。核心是 pull-based 长轮询——你的 PC 不需要公网 IP、不需要端口映射、不需要 VPN，只要能访问服务端就行。

**小夏：** 具体流程？

**智哥：** 五步。注册声明模型列表，30 秒心跳保活，长轮询阻塞式等任务，本地 Ollama 执行，提交结果。有个特别的设计——容错参数都是生产打磨过的。比如 complete_grace_secs 设了 60 秒宽限期：假设任务 deadline 2 分钟，节点在 1 分 59 秒算完了但还没来得及提交结果，这 60 秒给了"慢半拍"的节点补交作业的机会。

**小夏：** 怎么让请求路由到我的 PC？

**智哥：** 模型名前加 node: 前缀，比如 node:deepseek-chat，路由引擎检测到就走独立通道，直接查有没有在线节点能处理。

**小夏：** 那 PC 挂了会不会自动 fallback 到云端？

**智哥：** 不会。用户显式选了 node: 说明要用本地算力——有些数据不能上云，隐私和合规是刚需。自动 fallback 违背用户意图。

**小夏：** 路由到节点时信任客户端自报信息吗？

**智哥：** 不信任。NodeCapabilityIndex 在服务端验证所有条件——在线、没过期、没撤销、支持目标模型。全部服务端查库验证，不信客户端自报负载。

**小夏：** 启发？

**智哥：** NAT 穿透用 pull 最简单，别跟防火墙硬刚。用户明确选了本地算力就别偷偷 fallback。容错要给宽限，不能一刀切。

## 六、后置计费：不可变主账本（~1.5min）

<!-- PPT: 三不原则卡片 + rust_decimal vs f64 对比 + 三种结算状态 -->

**小夏：** 本地和云端都走 Gateway 执行，费用怎么算？

**智哥：** 三不原则。不参与路由，不预扣余额，不影响执行结果——即使计费失败，已返回的结果不会撤回。

**小夏：** 为什么要后置？预扣余额不是很自然吗？

**智哥：** 预扣余额在高并发下是性能杀手——每次请求都要查余额，还要处理并发扣款竞态。KeyCompute 把计费从热路径摘掉了。

**小夏：** 精度怎么保证？

**智哥：** 用高精度定点数计算金额，不用浮点数。浮点数 0.1 加 0.2 不等于 0.3，百万请求累积下来偏差巨大。流结束后一次性写入不可变主账本。

**小夏：** 请求只完成了一半呢？

**智哥：** 三种结算状态——完全成功、部分完成、上游错误。只完成一半，已产生的 token 仍被计费。长对话场景尤其重要。

**小夏：** 启发？

**智哥：** 把计费从关键路径拿掉，高并发不卡在"查余额"上。金额用定点数不用浮点。半完成的请求也要计费，不然用户白嫖。

## 总结（~1.5min）

<!-- PPT: 五原则总结页 -->

**小夏：** 好，六个模块都过了一遍。帮我们提炼几个能落地的原则？

**智哥：** 五个。

第一，设唯一执行层。对外请求只走一个出口，审计清晰监控集中。在你的平台里找出"谁都能发请求"的漏洞，堵上它。

第二，计费后置。别在入口查余额扣钱，流结束再算账，省掉锁竞争。

第三，路由和执行分离。路由只读可预测，健康数据闭环但不互相阻塞。冷却不是封禁，是暂时惩罚。

第四，类型驱动加编译隔离。模块可见性编译期强制架构约束，比代码规范靠谱得多。即使你不用 Rust，模块级 access control 也能实现类似效果。

第五，用户明确意图不要偷偷 override。选了本地算力就别 fallback 到云端，尊重用户选择比性能更重要。

这五个原则，KeyCompute 用 18 个模块的边界守护来实践。你不需要重写整个系统，只要在自己的网关里找一个点先落实第一条——先堵住那个谁都能发请求的漏洞。

**小夏：** KeyCompute 已经开源，GitHub 地址在视频简介，MIT 协议可以直接商用。如果你正在做 AI 网关或者算力调度平台，这五个原则能直接拿去用。我们下期再见。

**智哥：** 再见。
