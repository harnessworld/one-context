mechanism · prefix

前缀匹配机制到底是怎么工作的

缓存的是 token 序列上的中间状态；精确哈希判定一致——不是模糊匹配。

pipeline · hash → hit / miss
cache-key intuition
# 断点前每一位都要对齐
if hash(prefix_1..N) == stored_hash:
    reuse_kv_and_prefix_state()
else:
    recompute_from_mismatch()

ordering

顺序就非常重要

ordering · SYSTEM → TOOLS → DYNAMIC

工具定义顺序

system / tools / messages 顺序摇摆 → 前缀哈希立刻失效。

稳定块前置

最稳定的系统提示与工具模式锁在头部。

禁止中间插入

动态内容只追加尾部，勿插队污染前缀。

缓存友好 = 可预测的静态前缀 + 可控的尾部增量。
