## P0 问题详情（五维度分析）

### P0-1：terraform destroy "清空所有自动备份" 不符合 AWS 实际行为

| 维度 | 内容 |
|-----|------|
| **问题现象** | 讲稿声称 Agent 执行 terraform destroy 后"连所有自动备份也一起被清空" |
| **问题成因** | AWS RDS 自动备份有保留期（1-35天），terraform destroy 不负责删除 AWS 内部自动备份机制 |
| **问题影响** | 误导观众对 terraform destroy 威力范围的认知，降低可信度 |
| **定级依据** | 核心开场案例的事实性错误，影响全篇可信度，判定为 P0 |
| **解决方案** | 删除"连所有自动备份也一起被清空"，改为"由 Terraform 管理的所有基础设施资源" |
| **最新状态** | ✅ 已解决 |

### P0-2："Session 重放状态"机制描述错误

| 维度 | 内容 |
|-----|------|
| **问题现象** | 讲稿称"恢复时从 Session 重放状态" |
| **问题成因** | Session 是 append-only 事件日志（记录 Agent 行为序列），非文件系统/进程状态快照 |
| **问题影响** | 观众误解恢复机制，以为 Session 能"回放"完整执行环境 |
| **定级依据** | 核心架构概念的事实性错误，判定为 P0 |
| **解决方案** | 改为"恢复时 Harness 从 Session 读取历史事件重建推理上下文，重新 provision 新 Sandbox 继续" |
| **最新状态** | ✅ 已解决 |

### P0-3：n8n 案例归因谬误

| 维度 | 内容 |
|-----|------|
| **问题现象** | 用 Pyodide 应用层逃逸论证 Docker 容器隔离不够 |
| **问题成因** | Pyodide 是 JS/WASM 兼容运行时，非 OS 级沙箱；它的逃逸证明的是"应用层沙箱不够"，而非"Docker 不够" |
| **问题影响** | 逻辑链断裂，技术观众会质疑论证严谨性 |
| **定级依据** | 核心论证链的事实错误，影响 Doker→microVM 推论的合理性，判定为 P0 |
| **解决方案** | 将案例归因修正为"应用层沙箱（Pyodide/WASM）不等于 OS 级隔离"，如需论证容器共享内核风险，应引用容器逃逸 CVE |
| **最新状态** | ✅ 已解决 |

| **最新状态** | ✅ 已解决 |

### P0-5：冷启动与预热池混淆

| 维度 | 内容 |
|-----|------|
| **问题现象** | 笼统说"冷启动只要60毫秒" |
| **问题成因** | Firecracker 真冷启动 ~125ms，60ms 是预热池已分配 VM 的就绪时间 |
| **问题影响** | 误导观众以为 microVM 真冷启动就是 60ms |
| **定级依据** | 性能数据的事实性错误，判定为 P0 |
| **解决方案** | 明确区分冷启动和预热池，60ms 归因为预热池技术 |
| **最新状态** | ✅ 已解决 |
