# 【栈层】AI workload 落点

男：**推理服务**：Cloudflare Infire，公开定位就是 Rust 写的 LLM 推理引擎，聊的是边缘侧推理成本和延迟这类硬账。

男：**Tokenizer**：Hugging Face 的 **tokenizers**，很多团队第一次「被迫」碰 Rust，就是词元化这条链在 CPU 上太烫。
女：四层合一句：**Rust 香在「基础设施默认项」**，不是香在「替代你写 notebook」。
男：同意，别把叙事读反了。

# 【权威信源】一级背书

女：**NSA、CISA** 在 CSI 里推 memory safe languages，二〇二五年前后公开材料写「减少内存安全类漏洞」很直白——我们引用它，是说监管和甲乙方在换话语体系，不是喊你明天重写内核。
男：**云厂商工程博客**用来核对数字和场景前提，数字一定要**带回原文**，别在口播里编全网通用真理。
女：**LWN、Ars** 这类媒体补的是历史纵深——Rust 不是凭空来的，也不是包治百病。
男：所以画面上那句 **语境优先**，比金句重要。

# 【语言机制】机制三板斧

时长：约 100s

女：收束到语言机制，三板斧，别玄学。
男：**所有权加 borrow checker**：无 GC 的可预测资源，对延迟敏感、要控尾延迟的服务更友好。
女：**类型化错误**：失败模式钉在编译期或服务边界，长期维护往往比「运行时才炸」省心。
男：**Fearless 并发**：CPU 热路径不吃 Python GIL 那套约束，该并行就并行。
女：**边界**也要说清：**unsafe 是显式的**，**审计有抓手**——有 unsafe 不等于烂工程，而是你知道该审哪里。
男：三板斧回答的是「为什么基础设施团队愿意付 Rust 的学习税」，不是「为什么你必须抛弃 Python」。
女：别被带节奏，先理解机制再谈选型。
