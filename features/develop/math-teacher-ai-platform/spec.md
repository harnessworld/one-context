# 概述

当前 `FunctionCanvas` 已经具备较强的数学可视化能力，但它本质上仍是一个单点产品：围绕函数图像、曲面、矢量场等内容进行表达式输入与图形演示。

目标不是在它上面继续堆几个按钮，而是将其升级为一个更大平台中的**子产品**，作为“数学概念演示与可视化引擎”接入到面向数学教师全生命周期的 AI 平台中。

Phase 1 的核心任务不是一次性做完 AI 备课、AI 班级、AI 出题、AI 阅卷、AI 评分，而是**先把平台框架搭起来**：统一入口、统一信息架构、统一对象模型、统一导航与工作台，把 `FunctionCanvas` 安置到正确的位置，为后续模块扩展留出稳定骨架。

---

# 目标与非目标

## 目标

- 定义平台级产品定位：面向数学教师的 AI 工作平台，而不是单一函数绘图工具
- 明确 Phase 1 的产品边界：先搭建平台骨架，再逐步扩展能力
- 将 `FunctionCanvas` 重新定义为平台子产品 `Math Canvas`
- 设计平台一级导航、工作台结构、核心对象模型与模块关系
- 定义后续子产品的插槽位：`AI备课`、`AI班级`、`AI出题与组卷`、`作业与练习`、`AI阅卷`、`AI评分与讲评`、`教研成长`
- 明确 Phase 1 推荐实现落点：优先在现有 `FunctionCanvas` 仓库内演进为单仓平台壳

## 非目标

- Phase 1 不要求真正完成全流程教务系统、LMS、校务管理能力
- Phase 1 不要求一次性落地完整后端、租户体系、支付、商业化后台
- Phase 1 不要求完成高精度数学 OCR 阅卷与步骤级评分
- Phase 1 不要求沉淀完整数学知识图谱，只要求预留对象模型与接口位
- Phase 1 不要求把所有子产品都做成可用版本，只要求框架、入口和示范模块成立

# 用户与场景

**核心用户 1 — 一线数学教师**
需要在一个入口里完成备课、演示、布置练习、查看班级学情、讲评试卷，而不是在多个孤立工具间来回切换。

**核心用户 2 — 数学教研组长 / 备课组长**
需要共享课例、共建题库、沉淀讲评模板、对比班级表现，并逐步建立校本数学资源资产。

**核心用户 3 — 学校管理者**
需要看到平台是否能形成统一入口、统一数据资产和可控的 AI 使用边界，而不是多个不可治理的 AI 小工具。

**场景 A — 备课**
老师从平台首页进入 `AI备课工作台`，根据教材章节、知识点、班级水平生成教案草稿、课堂流程和互动建议，并直接调用 `Math Canvas` 生成函数或几何演示资源。

**场景 B — 课中演示**
老师进入 `Math Canvas`，把可视化结果作为本节课资源的一部分展示；后续平台内其他模块也可引用同一演示资产，而不是重复制作。

**场景 C — 课后练习与讲评**
老师使用 `AI出题与组卷` 生成练习，学生提交后由 `AI阅卷 / AI评分` 输出结果，再回流至平台中的班级画像与下一轮备课建议。

# 产品定义

## 平台一句话定义

一个服务数学教师全生命周期的 AI 工作平台：围绕**备课、授课、班级、作业、出题、阅卷、评分、讲评、教研成长**建立统一工作台；其中 `Math Canvas` 是平台内的数学可视化子产品。

## 一级子产品

- `首页 / 工作台总览`
- `AI备课`
- `Math Canvas`（由 `FunctionCanvas` 升级接入）
- `AI班级`
- `AI出题与组卷`
- `作业与练习`
- `AI阅卷`
- `AI评分与讲评`
- `教研成长`
- `资源中心`

## Phase 1 实际落地范围

- 平台壳：品牌、导航、路由、工作台首页
- `Math Canvas` 正式接入为一级子产品
- `AI备课` 建立入口页与基础流程骨架
- `资源中心` 建立统一数学资产概念（教案、可视化、题目集、试卷、讲评稿）
- `AI班级 / 阅卷 / 评分 / 教研` 先提供模块占位、对象模型与扩展接口
