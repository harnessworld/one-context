# 产品设想

1. **唯一真相（single truth）**：规则写在一处（`_shared_rules.py` 或等价声明式来源），Claude Code、Cursor、OpenClaw 适配器共享，不各自维护副本。
2. **正向生成**：`onecxt adapt` 从唯一真相生成各平台可用的规则文件，生成物标记为 `GENERATED`。
3. **脏检测与保护**：生成物可能被 AI agent 直接修改；通过 Git pre-commit hook 检测源头与生成物的偏差，自动重新生成，避免静默丢失改动。
4. **多工具同步**：不依赖各工具自身的 hook 能力（Cursor / OpenClaw 无此机制），收束到 Git 层——所有工具都绕不过的公共层。

---

# 概述

today：`meta/profiles.yaml` 存放结构化 profile（`behavior`、`output_style` 等），但「字段值 → 自然语言规则」写在 `packages/one-context/one_context/adapters/` 下 `PROFILE_RULES` 的 Python 列表里，且 `claude_code.py` / `cursor.py` / `openclaw.py` 各维护一份相同逻辑（13 条规则 × 3 = 39 条 FieldRule），与架构文档「Write shared meaning once. Adapt it many times.」存在落差。

本需求：将 `PROFILE_RULES` 提取为共享定义，适配器只负责 import + 渲染；通过 Git pre-commit hook 和 `onecxt adapt --check` 保障多工具场景下源头与生成物始终一致。

# 目标与非目标

## 目标

- **DRY**：三适配器共享同一套规则定义（`_shared_rules.py` 或等价模块），删除重复的 `PROFILE_RULES` 常量。
- **单一真相**：规则文案有一份可版本化、可 diff 的来源，`adapter_overrides` 仅在需要时特化。
- **生成物标记**：所有生成物文件头带 `GENERATED by onecxt adapt` 标记 + 源头指引，引导 agent 改源头而非生成物。
- **Git 层同步**：提供 pre-commit hook 脚本，commit 时自动 `onecxt adapt --all` + stage 生成物，确保任意工具编辑源头后生成物跟随更新。
- **CI 守护**：`onecxt adapt --check` 模式（只检查不写入，不一致则 exit 1），作为 CI pipeline 兜底。
- **脏检测**：adapt 执行时比较磁盘现有生成物与期望输出，不一致时在终端报告 diff（不静默覆盖）。
- **行为语义等价**：迁移后生成物与当前输出语义等价（golden test 快照覆盖），允许统一文案措辞。
- **文档**：`docs/architecture.md` 中明确「共享规则」与 `profiles.yaml`、`knowledge/` 的分工。

## 非目标

- 自动反向同步（自然语言 diff → 结构化字段不可逆，不做）。
- 重写 profile 继承 / mixin 语义（沿用现有合并逻辑）。
- 将 `knowledge/` 全文并入规则定义（长文仍走 knowledge 层）。
- OpenClaw 多配置文件布局设计（复杂度高，拆为独立需求）。
- 实时 file watcher（`onecxt watch`）——作为 P2 可选增强，不在本需求范围。

# 多工具同步保障（三层防线）

## 第一层：实时（仅 Claude Code）

现有 PostToolUse hook，编辑源头文件后秒级触发 `onecxt adapt --only claude_code`。
已实现，无需额外工作。

## 第二层：提交时（所有工具）

Git pre-commit hook：

```bash
#!/bin/sh
# .git/hooks/pre-commit 或通过 husky / pre-commit 框架安装
onecxt adapt --all
git add .claude/ .cursor/ .openclaw/
```

无论用什么工具编辑源头，commit 时一定同步所有生成物。

## 第三层：CI 兜底

```bash
onecxt adapt --all --check
# 只比较不写入，不一致 exit 1
```

即使有人 `--no-verify` 跳了 hook，CI 也拦住。

# 开放问题

## 共享规则格式

1. **Python vs YAML**：共享规则是纯 Python 模块（`_shared_rules.py`，直接定义 `FieldRule` 列表）还是 YAML manifest + loader？当前规则仅 13 条且只有开发者编辑，Python 更简单；如果未来非开发者需要编辑规则再引入 YAML。
2. **文案统一**：当前三适配器同一规则的措辞略有不同（如 claude_code 写 "Take a conservative approach…"，cursor 写 "Be conservative…"）。迁移时是否统一为一份文案 + 按需 `adapter_overrides`？
3. **`adapter_overrides` 归属**：特化逻辑（如 Claude Code 的 `placement: top`）全部写在共享模块里，还是各适配器在 import 后叠加？

## pre-commit hook 安装

4. **安装方式**：裸脚本（`scripts/pre-commit` + 手动 symlink）vs 集成 husky / pre-commit 框架？
5. **性能**：adapt 全量生成是否影响 commit 速度？是否需要增量检测（只在源头文件有 diff 时才触发）？

# 备注

- 本需求刻意不做「反向同步」（平台侧编辑自动回流唯一真相）。自然语言规则文案不可逆解析为结构化字段，试图实现只会引入脆弱性。脏检测 + pre-commit 覆盖已足够保障一致性。
- `onecxt watch`（file watcher 守护进程，监听源头变更后实时 adapt）作为 P2 增强，适用于多工具并行使用场景，不阻塞本需求交付。
