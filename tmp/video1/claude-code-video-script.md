# 《Claude Code 源码泄露，顶级 AI 工具到底怎么写的？》

**角色：**
- **老周**（技术专家，资深架构师风格，语速沉稳）
- **小林**（好奇的开发者，有一定基础但对架构设计不深）

---

**【开场钩子 - 40秒】**

**小林：** 老周，我跟你说个事，Claude Code 的完整源码泄露了！

**老周：** 我知道。npm 包里的 source map 没删干净，整个 TypeScript 源码几乎被 1:1 还原出来了。

**小林：** 但我看完后整个人都傻了——Anthropic 这种估值几百亿美金的公司，它最核心产品的全局状态管理，竟然只写了 35 行代码？

**老周：** 还不止呢。它里边有个黑科技，能在你按下回车之前，就把你要做的事提前做完了。而且源码注释里还记了一个大事故：一个随机生成的 UUID，居然让他们的 API 成本直接翻了 12 倍！

**小林：** 这全是真金白银砸出来的实战避坑指南！**各位还没点关注的赶紧点个关注点赞收藏**，今天老周带你把这套顶级 AI 工具的架构底座一层层拆开，今天聊的每一个点，你都能直接搬到自己的项目里复用。

---

**【第一部分：35行代码干翻状态管理库 - 90秒】**

**小林：** 它的全局状态管理，核心代码只有 35 行，文件叫 `state/store.ts`。

**老周：** 对，一个极简的 Pub-Sub Store。createStore 接收初始状态，返回三个方法：getState、setState、subscribe。setState 里用 `Object.is` 做引用比较，相同就跳过，不同就通知所有订阅者。

**小林：** 等一下，这不就是 Zustand 干的事吗？为什么不直接用 Zustand？

**老周：** 这就是关键问题了。Zustand 本身也很小，大概 1KB，但它带了 middleware、devtools、immer 这些生态。Claude Code 是个 CLI 工具，跑在终端里，它不需要这些。35 行代码，零依赖，配合 React 18 的 `useSyncExternalStore` 就能实现细粒度订阅。

**小林：** 细粒度是指？

**老周：** 就是你消费状态的时候，传一个 selector 函数，只取你需要的那个字段。这个字段没变，你的组件就不会重新渲染。源码里还专门加了注释——禁止在 selector 里返回整个 state 对象，因为每次都会创建新引用，会导致无限渲染。

**小林：** 这个设计其实任何 React 项目都能直接抄过去用。

**老周：** 没错。中小型项目根本不需要引一个状态管理库，35 行自己写一个，控制力更强，调试也更方便。

---

**【第二部分：推测执行引擎 - 120秒】**

**小林：** 好，第二个我想聊的是推测执行。这个 `speculation.ts` 将近一千行代码，是我觉得整个项目里最惊艳的设计。

**老周：** 这个确实厉害。简单说就是——模型回答完之后，系统会预判你接下来最可能说什么，然后在你还没按回车之前，就已经在后台把活干了。

**小林：** 具体怎么做到的？

**老周：** 三步。第一步，模型响应结束后，系统生成一个"下一步建议"，就是猜你可能会点"继续"或者说"好的，执行吧"。第二步，它在系统临时目录里创建一个 overlay 目录，相当于一个沙盒。然后 fork 出一个子代理，在这个 overlay 里执行推测的操作。第三步，如果你确认了，就把 overlay 里的文件变更合并回主工作目录。如果你说的不是它猜的那样，直接丢弃，零副作用。

**小林：** 那安全性怎么保证？万一它在后台偷偷跑了个危险命令呢？

**老周：** 这是最精妙的地方。它把工具分成了三类：只读工具，比如 Read、Glob、Grep，这些直接在主目录执行，因为本来就不改东西；写入工具，比如 Edit、Write，这些必须走 overlay，写到沙盒里；而 Bash 命令？直接不执行，遇到就停止推测。

**小林：** 所以它的安全边界很清晰——读操作放行，写操作隔离，命令执行禁止。

**老周：** 对。还有两个硬限制，最多推测 20 轮，最多 100 条消息。防止失控。

**小林：** 这个"预判→沙盒执行→确认合并"的模式，其实不光 AI 工具能用。IDE 自动补全预执行、数据库迁移预演，都是同一个思路。

**老周：** 你理解得很到位。

---

**【第三部分：一个 UUID 引发的 12 倍成本 - 90秒】**

**小林：** 第三个点我觉得特别有教训意义。源码注释里记录了一个真实事故——一个随机 UUID 导致 API 成本暴涨 12 倍。

**老周：** 这个背景是这样的。Claude 的 API 支持 Prompt Cache，就是如果你连续多次调用，前面的系统提示和工具描述都一样的话，服务端可以复用之前的计算结果，不用重新处理。这能省巨量的 token 成本。

**小林：** 然后呢？

**老周：** 问题出在设置文件的临时路径上。这个路径会出现在 Bash 工具的沙箱配置里，而沙箱配置是工具描述的一部分，工具描述会发给 API。之前的写法是每次启动子进程都生成一个随机 UUID 作为文件名。结果呢，每次调用 API 时工具描述都不一样，缓存前缀永远匹配不上，每次都是全量计算。

**小林：** 所以每次请求的输入 token 都按全价收费？

**老周：** 对。修复方法很简单，把随机 UUID 换成内容哈希。设置内容没变，路径就不变，缓存就能命中。一行代码的改动，成本直降到原来的十二分之一。

**小林：** 这个教训太实际了。所有调 LLM API 的团队都应该检查一下——你发给 API 的内容里，有没有混入随机值？时间戳、UUID、临时路径，这些都是缓存杀手。

**老周：** 源码里还专门写了一个 `promptCacheBreakDetection.ts`，七百多行，用三组哈希追踪系统提示、工具 schema 和缓存标记的变化，一旦检测到缓存被打破就告警。这才是工程化的做法。

---

**【第四部分：Bash 执行的三层安全防线 - 100秒】**

**小林：** 第四个点聊安全。BashTool 相关的代码加起来超过 5000 行，是整个项目里最重的模块。

**老周：** 这很合理。AI 帮你执行 shell 命令，这是整个系统里风险最高的操作。它用了三层防御。

**小林：** 哪三层？

**老周：** 第一层，静态校验。23 种检查规则，每种都有唯一数字 ID 方便追踪。包括不完整命令检测、混淆参数检测、shell 元字符检测、IFS 注入检测、控制字符检测等等。这一层不执行任何东西，纯粹做文本分析。

**小林：** 第二层呢？

**老周：** 第二层，权限检查加复合命令拆解。这里有一个真实的安全修复案例特别有意思。假设用户写了 `echo hello && rm -rf /`，如果你只检查整条命令的前缀，它以 echo 开头，看起来是安全的。所以修复方案是把复合命令拆开，按 `&&`、`||`、`;` 分割后逐个检查。还有一个管道绕过的案例——管道每段单独看都合规，但组合起来的重定向行为没被检查到。

**小林：** 所以安全检查不能只看表面，要拆到原子级别。

**老周：** 对。第三层是系统级沙箱。Linux 上用 bwrap，macOS 上用 sandbox-exec。文件系统和网络都有约束，违规操作会被自动标注到 stderr。

**小林：** 三层叠加：先过文本筛子，再过权限规则，最后上操作系统级隔离。

**老周：** 这就是纵深防御。任何让 AI 执行代码的产品，都应该认真研究这套模型。

---

**【第五部分：API 防雪崩 - 70秒】**

**小林：** 最后聊一个后端同学都会关心的——它的 API 重试策略。

**老周：** 这里有一个设计特别值得学。当服务端返回 529 过载错误时，它会区分请求来源。如果是前台用户的请求，正常指数退避重试。但如果是后台任务的请求，比如子代理、批量操作——直接丢弃，不重试。

**小林：** 为什么？

**老周：** 防雪崩。服务端已经过载了，后台任务的重试只会让情况更糟。前台请求关系到用户体验，值得重试。后台任务晚几分钟执行没人在意。这叫"按请求源分级"的重试策略。

**小林：** 还有一个 Fast Mode 的降级也很聪明。429 限流时，如果 retry-after 小于 20 秒，就保持 Fast Mode 短暂等一下再试，因为这样能保住 prompt cache。但如果要等超过 20 秒，cache 大概率已经过期了，那就降级回标准模式，不等了。

**老周：** 对，它始终在做一个权衡——保住缓存的收益和等待的成本之间的平衡。
