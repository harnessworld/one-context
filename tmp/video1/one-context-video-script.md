# 《用 AI 写代码的人都踩过这些坑，这个开源项目一次全解决》

**角色：**
- **老周**（技术专家，资深架构师风格，语速沉稳）
- **小林**（好奇的开发者，有一定基础但对多仓协作和 AI 工具链不深）

---

**【开场钩子 - 40秒】**

**小林：** 老周，你有没有这种感觉——Cursor、Claude Code 这些 AI 工具单个用都挺好，但一旦项目大了、仓库多了，它们就开始各说各话？

**老周：** 太常见了。你在 Cursor 里配了一套规则，Claude Code 里又得抄一遍，OpenClaw 又是另一套格式。改一条规范，三个地方都得改，漏一个就出事。

**小林：** 更烦的是，这些工具都只能看到当前这一个仓库。可我们实际干活，前端、后端、基础设施、文档，经常是四五个仓库一起动。AI 完全不知道隔壁仓库在干嘛。

**老周：** 今天聊的这个开源项目 one-context，就是专门解决这个问题的。它不是又一个 AI 工具，而是一个**控制面**——在所有 AI 工具的上面，统一管理上下文、规范和协作流程。**感兴趣的先点个关注**，我们一层层拆。

---

**【第一部分：写一次，适配所有工具 - 100秒】**

**小林：** 先说最让我头疼的那个问题。我现在项目里同时用 Cursor 和 Claude Code，Cursor 要写 `.cursorrules`，Claude Code 要写 `CLAUDE.md`，两边格式完全不一样。

**老周：** one-context 的核心理念就六个字——**写一次，到处用**。你把编码规范、项目知识、行为偏好，全写在一个中立的地方，然后跑一条命令 `onecxt adapt`，它自动给你生成每个工具的原生配置。

**小林：** 具体怎么做的？

**老周：** 它有一个 Profile 系统。你声明你的行为偏好，比如"改代码前先出方案让我确认"、"保守一点、优先最小改动"、"每次改完跑一遍测试"。这些是语义化的字段，不是写给某个工具的提示词。

**小林：** 然后呢？

**老周：** 然后 Adapter 层做翻译。每个工具有一个适配器，大概六十行代码。它把这些语义字段翻译成那个工具能理解的自然语言规则。Cursor 适配器生成 `.mdc` 文件，Claude Code 适配器生成 `CLAUDE.md` 和 agent 文件，OpenClaw 适配器生成 JSON。翻译引擎是一个声明式的规则匹配器——字段路径、匹配值、输出句子，没有模板语言，纯 Python，零额外依赖。

**小林：** 所以我改一条规范，所有工具的配置自动同步？

**老周：** 对。而且你要接一个新工具，只需要写一个六十行的适配器就行了。这才是正确的工程化做法——**配置是派生物，不是源头**。

---

**【第二部分：打破单仓围墙 - 90秒】**

**小林：** 第二个痛点。我现在有个微服务项目，五个仓库。我在 Cursor 里写后端代码的时候，它完全不知道前端那边接口已经改了。

**老周：** 这就是所有 AI 编码工具的盲区——它们天然是单仓视角。one-context 用一个叫 Workspace 的概念来解决这个问题。

**小林：** Workspace 不就是文件夹吗？

**老周：** 不是。Workspace 是一个逻辑视图——你可以把五个不同仓库的代码按任务或主题组合在一起。比如你在做"用户认证"这个特性，你就建一个 workspace，把后端服务、前端应用、基础设施这三个仓库都拉进来，再挂上相关的编码规范和领域知识。

**小林：** 这些仓库本身还是独立的 Git 仓库？

**老周：** 完全独立。one-context 不是 monorepo 工具，仓库该怎么管还怎么管。它只是在上面加了一层声明式的注册表——`meta/repos.yaml` 记录所有仓库，用稳定的 id 引用，不管仓库搬到哪个目录、改了什么 URL，引用都不会断。

**小林：** 所以 AI 工具通过 workspace 就能"看到"多个仓库的上下文了？

**老周：** 对。`onecxt adapt` 会把 workspace 涉及的所有仓库的知识、规范、接口约定，全部注入到工具的配置里。AI 不再是单仓盲人了。

---

**【第三部分：七个 Agent 分工协作 - 120秒】**

**小林：** 第三个点我觉得特别有意思——它定义了一套多 Agent 协作框架。

**老周：** 对，七个标准角色，覆盖软件研发全生命周期。PM 管需求，Architect 管技术方案，Dev 写代码，QA 做测试，SRE 管发布，还有一个 Reviewer 专门做方案评审，一个 Knowledge Keeper 维护知识库。

**小林：** 但 AI 工具不是一个人吗？为什么要分这么多角色？

**老周：** 这是这个项目最核心的设计思想——**每个文件有且只有一个 owner**。PM 只能写 `spec.md`，Architect 只能写 `tech_design.md`，Dev 只能改代码，QA 只能写测试报告。谁都不能越界。

**小林：** 为什么要这么严格？

**老周：** 两个原因。第一，防止 AI 做它不该做的事。你让 AI 帮你写技术方案，它写着写着突然开始改代码了——这种情况你遇到过吧？有了 artifact ownership，它做不了。第二，可追溯。每个文件是谁产出的、基于什么输入、在哪个阶段产出的，全部清清楚楚。

**小林：** 那它们之间怎么协作？

**老周：** 显式的文件交接。PM 写完 spec.md，Architect 读 spec 写 tech_design.md，Dev 读 design 建 worktree 写代码，QA 拿着 spec 和代码写测试报告，SRE 读 deliver.md 执行发布。每一步的输入输出都是文件，全部可以 git 追踪。

**小林：** 所以它不是让多个 AI 实例同时跑，而是定义了一套"接力"的流程？

**老周：** 对。这其实就是软件工程里"关注点分离"的思想，只不过它把这个思想应用到了 AI Agent 的协作上。你可以用同一个 Claude Code 实例，切换不同的 agent 角色来工作，每个角色看到的知识、行为规则、权限范围都不一样。

---

**【第四部分：知识不丢、规范不漂 - 90秒】**

**小林：** 第四个点，知识管理。老实说我现在的项目规范散落在 Wiki、聊天记录、各种 `.rules` 文件里，根本没人维护。

**老周：** 这是所有团队都会遇到的问题——知识漂移。代码改了，文档没改，AI 按过时的规范干活，结果还不如不用 AI。

**小林：** one-context 怎么解决的？

**老周：** 它建了一个分层的知识体系。`knowledge/standards/` 放编码规范和工程约定，`knowledge/playbooks/` 放操作手册——比如"如何添加一个跨仓特性"的分步指南，`knowledge/references/` 放外部资源索引。全是 Markdown，全是人类可读的，不掺任何工具特有的语法。

**小林：** 这些知识怎么跟 AI 工具打通？

**老周：** Workspace 和 Agent 定义里都可以引用知识文件。跑 `onecxt adapt` 的时候，适配器会把这些知识注入到工具配置里——支持 `@file` 引用的工具用引用，不支持的就直接内联全文。

**小林：** 那如果知识过时了呢？

**老周：** 这就是 Knowledge Keeper agent 的职责。它的工作就是检测代码和知识之间的偏差，发现不一致就提 PR 更新。而且过时的标准不是直接删，是归档到 `knowledge/archive/`，保留历史。

**小林：** 所以知识是活的，不是写完就扔在那的。

**老周：** 对。而且因为知识和工具配置解耦了，你改了一条标准，所有工具的行为自动跟着变。这才叫**知识驱动的 AI 协作**。

---

**【第五部分：零依赖的工程哲学 - 70秒】**

**小林：** 最后一个点让我挺意外的——这个项目除了 PyYAML，几乎零外部依赖。

**老周：** 这是有意为之的。它的规则引擎没有用 Jinja2，Profile 系统没有用模板语言，同步用的是标准库 subprocess 调 git，CLI 也是纯标准库。

**小林：** 为什么要这么克制？

**老周：** 因为 one-context 的定位是基础设施。它要跑在各种 Python 环境上——3.10 到 3.13，macOS、Linux、Windows 都要支持。依赖越少，出问题的概率越低，安装越简单。而且它的核心逻辑本身就不复杂——注册表加载、字段匹配、文件生成，这些用标准库完全够了。

**小林：** 那个声明式规则引擎我看了，确实挺巧妙。不用写模板，只要声明"哪个字段等于什么值就输出什么句子"。

**老周：** 对。这个设计让写适配器变得非常简单——你不需要理解模板渲染，不需要学 DSL，就是声明字段规则，引擎帮你匹配。六十行代码就能接入一个新的 AI 工具，这才是好的抽象。

---

**【收尾总结 - 40秒】**

**小林：** 总结一下今天聊的五个点：写一次适配所有工具的 Adapter 模式、打破单仓围墙的 Workspace 机制、七个 Agent 分工协作的 artifact ownership、知识驱动的活文档体系、还有零依赖的工程克制。

**老周：** one-context 做的事情，本质上是把软件工程里那些已经验证过的思想——关注点分离、单一职责、配置即代码、声明式优于命令式——重新应用到了 AI 工具链上。它不发明新理论，它解决的是实际干活时每天都遇到的问题：规范不同步、上下文断裂、AI 越权操作、知识散落。

**小林：** 项目是 MIT 开源的，Python 写的，安装就一条 pip 命令。链接放评论区了，特别是用多个 AI 工具、管多个仓库的同学，强烈建议去看看。我们下期见。
