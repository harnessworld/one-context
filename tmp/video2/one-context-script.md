# one-context 项目口播逐字稿（纯录制版）
**角色**：小林、老周
**总时长**：3分20秒
**核心禁忌**：不提MCP实现多端适配、智能体6个、命令`onecxt`、仓库`harnessworld/one-context`、MIT开源

---

## 一、黄金开场
小林：老周！个人写代码都快了，人和人、仓和仓之间的缝怎么越裂越大？
老周：有篇文章说得好：AI把每个人拉成高速公路，接缝还是乡间小道。你这边Cursor、Claude Code单用都香，一上多仓、多工具——规范抄三遍、上下文断档，很正常。
小林：对！改一条规范几个工具抄几遍；后端改接口，前端AI跟瞎子一样；还有人越权改代码——烂尾！
老周：今天这个项目one-context，开源、本地优先。把registry、知识、需求链收进伞形仓——一处定义，处处同步；不是又一个模型，是共享上下文层。想听怎么落地的，先关注。

## 二、写一次，处处同步
小林：最烦：每家工具一套格式，.cursorrules、CLAUDE.md……改一条规范，像打地鼠！
老周：把「真理」放回umbrella：规范、profile、智能体写在meta/、knowledge/——权威在清单，各工具配置文件只是派生物。
小林：说人话：怎么落到磁盘上？
老周：一条`onecxt adapt`：声明式规则引擎把同一份语义编译成Cursor .mdc、Claude适配器Markdown、OpenClaw JSON，是本地CLI直接写文件，不是运行时跑出来的。
老周：新工具？大约60行适配器就能接入，别再复制粘贴全世界。
小林：多端抄规范抄吐的，扣1

## 三、跨仓 Workspace
小林：死穴：微服务好几个仓，SDD写得再漂亮，AI拆需求也像盲人摸象——看得见契约，摸不准实现里的隐含约定！
老周：工具默认都是单仓视角；one-context用Workspace，不是多建个文件夹，是在workspaces.yaml里声明，这个任务要哪几个仓、什么视角。
小林：打住——必须全员搞物理monorepo才能治？我们根本搬不动啊！
老周：不必。各仓还是独立Git，伞形仓只做登记对齐，`onecxt sync`拉工作副本，是逻辑全景，不是把代码历史硬拧成一棵树。
老周：再给模型一兜上下文，`onecxt context export`整包带走；内容太长就按token预算压缩，先保证事实不断裂。

## 四、6角色与产物链
小林：还有更绝的！我让AI写方案，它写着写着直接改我代码，完全越权！
老周：one-context用Agent框架，6个标准角色——PM、架构、开发、QA、SRE、knowledge-keeper，每个都绑定profile、knowledge和owns产物所有权。
小林：owns是啥？
老周：比如PM盯spec和需求索引，架构盯tech_design，QA盯测试、MR报告。伞形需求在features/里形成一条链：spec到设计再到测试交付，阶段产物分开、可审计，不是糊在聊天里一团乱。
老周：这就是把软件工程里的关注点分离，落到AI协作上。
小林：被AI越权坑过的，扣2

## 五、知识层与漂移
小林：规范散在Wiki、群、各种文件里，新人找不到，老人改了忘同步，AI还在用半年前的老规矩！
老周：knowledge/里全是工具中立的Markdown，标准、手册、提示词，给人看也给模型用，和workspace、profile、智能体绑定，改一处，执行一次adapt，各端一起更新。
老周：knowledge-keeper这个角色，专职维护标准和手册，不是魔法自动抓全网，是工程上把谁维护知识写进角色和权限里，全程可追溯。
老周：再配合`onecxt doctor`校验清单，元数据出错先拦一道。

## 六、工程克制
小林：这种基建我最怕，装一堆依赖、环境搞成地狱……
老周：我们依赖故意做的很克制：除了PyYAML解析清单，核心大量用标准库；适配层没有模板引擎，没花里胡哨的构建。一条pip命令，Windows、Mac、Linux全平台跑。
老周：本地优先，无云、无遥测绑架，代码和知识绝不离开你的机器。

## 七、收尾
小林：总结一下：一套语义多端适配、跨仓Workspace+sync、六角色权限+需求链、知识层+校验、轻依赖本地跑！
老周：说白了：把registry、knowledge、features、适配器、导出，收成一块共享上下文，塑造路径让对齐比将就更省力，不是只生成几份配置文件就完事。
小林：MIT开源，GitHub仓库harnessworld/one-context，克隆后执行pip install -e ./packages/one-context，`onecxt doctor`直接跑起来。多仓多工具的开发者，赶紧去试！关注我，下期手把手教adapt和sync上手，下期见！
